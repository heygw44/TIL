# TIL: N+1 문제

## 1. N+1 문제의 실체
 
### 1.1 Hibernate 공식 정의 (유저 가이드 Fetching 챕터)
 
> "Performs a separate SQL select to load the data. This is the strategy generally termed N+1."
 
즉 "연관 데이터를 별도 SELECT로 가져오는 전략"이 N+1의 원인이다. 한 번의 초기 쿼리 뒤에, 프록시/컬렉션 접근 시마다 별도 SELECT가 나가는 구조이다.
 
### 1.2 발생 조건 — 2가지가 동시에
 
1. **LAZY 로딩** (기본값인 경우가 많음)
   - `@OneToMany`, `@ManyToMany` → 기본 LAZY
   - `@ManyToOne`, `@OneToOne` → 기본 EAGER (Vlad는 "terrible default"라 부름)
2. **프록시/컬렉션 접근** (for 루프, stream, getter 호출)
### 1.3 재현 코드
 
```java
// 도메인: Article 1:N Comment
@Entity
public class Article {
    @Id @GeneratedValue
    private Long id;
 
    @OneToMany(mappedBy = "article", fetch = FetchType.LAZY)
    private List<Comment> comments = new ArrayList<>();
}
 
// 서비스
List<Article> articles = articleRepository.findAll();  // [1] SELECT article
for (Article a : articles) {
    a.getComments().size();                            // [N] SELECT comment per article
}
```
 
Article이 100건이면 **101 쿼리** 발생.
 
### 1.4 눈으로 보기 — 로깅 설정
 
```yaml
# application.yml
logging:
  level:
    org.hibernate.SQL: DEBUG               # 생성된 SQL 찍기
    org.hibernate.orm.jdbc.bind: TRACE     # 바인딩 파라미터까지 (Hibernate 6+)
spring:
  jpa:
    properties:
      hibernate:
        generate_statistics: true          # 세션 끝나면 "N queries" 통계 출력
        format_sql: true
```
 
> **배운 점**: `generate_statistics`가 N+1 탐지에 제일 빠르다. 로그를 눈으로 세는 것보다 세션 통계에서 `queries executed: 101`이 바로 뜨기 때문.
 
### 1.5 숨은 함정 — `@ManyToOne`도 N+1 낸다
 
JPQL로 자식(Comment)을 먼저 조회 후 `comment.getArticle()` 접근 시
 
> "The entity query fetch policy cannot be overridden, so Hibernate requires a secondary select."
> *— Hibernate User Guide*
 
즉, `@ManyToOne(fetch = EAGER)`여도 JPQL 쿼리는 이걸 재정의하지 못하고 별도 SELECT를 발행. 이게 `@ManyToOne`에서도 N+1이 터지는 메커니즘.

## 2. 해결책 — Fetch Join
 
### 2.1 작동 원리
 
JPQL의 `join fetch`는 연관을 **같은 SELECT의 OUTER JOIN으로** 가져옴.
 
```java
@Query("select distinct a from Article a join fetch a.comments")
List<Article> findAllWithComments();
```
 
SQL
```sql
SELECT a.*, c.*
FROM article a
LEFT OUTER JOIN comment c ON a.id = c.article_id;
```
 
쿼리 1방으로 끝낼 수 있다.
 
### 2.2 `distinct`는 왜 붙이나
 
Article 1건 + Comment 5건이면 JOIN 결과는 5행 → JPA 결과는 같은 Article 레퍼런스가 5번 들어감. `distinct`는 **JPA 레벨에서만 중복 제거**(같은 엔티티 레퍼런스는 하나로). SQL DISTINCT와는 의미가 다르다.
 
### 2.3 컬렉션 Fetch Join + 페이징
 
```java
em.createQuery("select p from Post p left join fetch p.comments", Post.class)
  .setMaxResults(5)
  .getResultList();
```
 
로그
```
WARN : HHH000104: firstResult/maxResults specified with collection fetch; applying in memory!
```
 
**왜?** Vlad의 설명이 가장 정확하다.
 
> "Hibernate cannot just truncate the result set using SQL-level pagination. If it did, the result set would be truncated in the middle of the PostComments rows, returning a Post with just a subset of comments. Because Hibernate favors consistency, it fetches the entire result set and does the pagination in memory."
 
핵심은 다음과 같다.
- OUTER JOIN으로 **Post × Comment 카티션 곱** 발생
- SQL `LIMIT 5`는 "Post 5개"가 아니라 "조인된 **5행**"을 의미
- SQL 페이징이 의미를 잃으므로 Hibernate는 **전체 결과셋을 다 긁어온 후** 메모리에서 페이징
- 데이터가 10만건이면 즉시 OOM
### 2.4 Hibernate 6부터는 경고 → 예외
 
Hibernate 6.0 이후 이 동작은 **WARN이 아니라 RuntimeException**으로 격상됨. 이제 "경고는 무시하자"가 통하지 않는다.
 
### 2.5 Fetch Join의 진짜 한계 요약
 
| 상황 | 가능? |
|---|---|
| ToOne 연관 (Article → Author) | 가능 |
| 컬렉션 1개 (Article.comments) | 가능하지만 페이징 불가 |
| 컬렉션 2개 이상 (Article.comments + Article.tags) | 불가능 (`MultipleBagFetchException`) |

## 3. `MultipleBagFetchException` — 컬렉션 2개 이상 Fetch Join 금지
 
### 3.1 에러 메시지
 
```
org.hibernate.loader.MultipleBagFetchException:
cannot simultaneously fetch multiple bags: [Post.comments, Post.tags]
```
 
**주의**: 이 에러는 쿼리 실행이 아니라 **애플리케이션 빌드/세션팩토리 생성 단계에서** 던져짐. 즉 런타임 트래픽이 없어도 앱 시작 자체가 실패.
 
### 3.2 Hibernate 용어: Bag vs List vs Set
 
| 타입 | Java 매핑 | 순서 | 중복 |
|---|---|---|---|
| **Bag** | `List` + 순서 어노테이션 없음 | 없음 | 허용 |
| **List** | `List` + `@OrderColumn` / `@OrderBy` | 있음 | 허용 |
| **Set** | `Set` | 없음 | 제거 |
 
> Java의 `List`와 Hibernate의 내부 타입 "List"가 다름. 그냥 `List<Comment>`라고 쓰면 내부적으로는 **Bag**으로 취급된다.
 
### 3.3 왜 Bag 2개는 동시 fetch가 금지되는가
 
Vlad의 설명
 
> "MultipleBagFetchException is thrown because duplicates can occur, and the unordered List, which is called a bag, is not supposed to remove duplicates."
 
Post × Comment(3) × Tag(2) = **6행의 카티션 곱**이 SQL에서 나옴. Hibernate 입장에서는
- "이 중복이 JOIN 때문에 생긴 가짜 중복인가?"
- "아니면 원본 데이터에 실제로 있는 진짜 중복인가?"
를 구분할 기준이 없음. 순서 정보도 없고 Set처럼 중복 제거도 못하니까.
### 3.4 흔한 잘못된 해결 — "`List` → `Set`"
 
많은 블로그가 권장하지만 **반쪽짜리 해결**. 예외는 사라지지만 카티션 곱 SQL은 그대로 나감.
 
> "There are so many blog posts, videos, books, and forum answers providing the wrong solution."
> *— Vlad Mihalcea*
 
Post 100 × Comment 10 × Tag 5 = **5,000행** → 네트워크·메모리 낭비.
 
### 3.5 올바른 해결 — 쿼리 2개로 나누기
 
```java
// 쿼리 1: comments만
List<Post> posts = em.createQuery("""
    select p from Post p
    left join fetch p.comments
    where p.id between :minId and :maxId
    """, Post.class)
    .setParameter("minId", 1L)
    .setParameter("maxId", 50L)
    .getResultList();
 
// 쿼리 2: 같은 Post들에 tags만
posts = em.createQuery("""
    select distinct p from Post p
    left join fetch p.tags t
    where p in :posts
    """, Post.class)
    .setParameter("posts", posts)
    .getResultList();
```
 
**작동 원리**: JPA의 1차 캐시(Persistence Context). 같은 Post ID에 대해서는 EntityManager 안에 **단 하나의 인스턴스**만 존재. 두 번째 쿼리 결과가 첫 번째 쿼리 결과의 `tags` 컬렉션을 그대로 채움. 결국 한 Post 인스턴스에 comments와 tags 모두 초기화됨.
 
> **깨달은 점**: 이 해결법은 "1차 캐시 특성"을 응용한 것. JPA의 Persistence Context가 어떻게 동작하는지 이해해야 납득이 감.

## 4. 해결책 ② — `@EntityGraph`
 
### 4.1 Fetch Join과의 차이 정리
 
| 항목 | Fetch Join | `@EntityGraph` |
|---|---|---|
| 정의 위치 | JPQL 문자열 안 | 어노테이션 (엔티티 or 리포지토리) |
| 재사용성 | 쿼리마다 복붙 | 하나 정의, 여러 메서드 적용 |
| 런타임 교체 | 불가 | 가능 (`FETCH` vs `LOAD`) |
| 생성 SQL | OUTER JOIN | OUTER JOIN (**결국 같음**) |
| 페이징 함정 | 있음 | 그대로 있음 |
| MultipleBag 함정 | 있음 | 그대로 있음 |
 
**핵심 인사이트**: `@EntityGraph`는 Fetch Join의 "대체"가 아니라 "재사용 가능한 선언 방식"일 뿐. **SQL 레벨에서는 동일한 OUTER JOIN**이 생성되므로 카티션 곱 문제는 똑같이 겪음.
 
### 4.2 사용 방법
 
**(1) attributePaths — 제일 흔한 방식**
```java
public interface PostRepository extends JpaRepository<Post, Long> {
    @EntityGraph(attributePaths = {"author", "comments"})
    List<Post> findAll();
}
```
 
**(2) NamedEntityGraph — 여러 곳에서 재사용**
```java
@Entity
@NamedEntityGraph(
    name = "Post.withAuthorAndComments",
    attributeNodes = {
        @NamedAttributeNode("author"),
        @NamedAttributeNode("comments")
    }
)
public class Post { ... }
 
// Repository
@EntityGraph(value = "Post.withAuthorAndComments")
Optional<Post> findById(Long id);
```
 
### 4.3 `FETCH` vs `LOAD` 타입
 
JPA 스펙 힌트
- `jakarta.persistence.fetchgraph` = `EntityGraphType.FETCH` (기본)
- `jakarta.persistence.loadgraph` = `EntityGraphType.LOAD`

| 타입 | 지정한 속성 | 지정 안한 속성 |
|---|---|---|
| `FETCH` | EAGER로 로딩 | **LAZY로 취급** |
| `LOAD` | EAGER로 로딩 | **매핑의 fetch 타입 따름** |
 
> **Hibernate 특이점**: JPA 스펙과 달리 Hibernate는 `FETCH` 타입에서도 정적으로 EAGER 설정된 속성을 함께 로드된다.
 
### 4.4 올바른 사용 패턴 (실무)
 
**ToOne 연관만 EntityGraph**로 가져오고, 컬렉션은 batch fetch로 처리하는 게 가장 적절하다.
 
```java
public interface PostRepository extends JpaRepository<Post, Long> {
    // author는 @ManyToOne → JOIN해도 카티션 곱 없음
    @EntityGraph(attributePaths = {"author", "category"})
    Page<Post> findAll(Pageable pageable);
}
```
 
`comments`는 LAZY로 두고 `default_batch_fetch_size`로 해결 → **페이징 + ToOne + 컬렉션 3개를 한번에 최적화**.

## 5. 해결책 ③ — `hibernate.default_batch_fetch_size`
 
### 5.1 핵심 메커니즘
 
LAZY 프록시 하나에 접근하면, **같은 세션에 있는 아직 초기화되지 않은 다른 프록시들까지 묶어서** `WHERE id IN (?, ?, ...)` 한 방에 로드.
 
Hibernate 공식 유저 가이드
> "Performs a separate SQL select to load a number of related data items using an IN-restriction as part of the SQL WHERE-clause based on a batch size."
 
### 5.2 설정
 
```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100  # 전역
```
 
또는 엔티티별 오버라이드
```java
@OneToMany(mappedBy = "post")
@BatchSize(size = 100)   // 이 컬렉션만 덮어쓰기
private List<Comment> comments = new ArrayList<>();
```
 
### 5.3 쿼리 변화 시연
 
**Before** (설정 없음): Post 100개 순회 시
```sql
select * from post;
select * from comment where post_id = 1;
select * from comment where post_id = 2;
...
select * from comment where post_id = 100;
-- 총 101 쿼리
```
 
**After** (`batch_fetch_size = 100`):
```sql
select * from post;
select * from comment where post_id in (1, 2, 3, ..., 100);
-- 총 2 쿼리
```
 
### 5.4 BatchFetchStyle 내부 알고리즘
 
Hibernate 유저 가이드에 나오는 3가지
 
- **`LEGACY`**: 미리 계산된 배치 크기 배열 `[25, 12, 10, 9, 8, ...]`에서 적절한 걸 선택. 최악의 경우 여러 번 쿼리.
- **`PADDED`**: 남은 ID가 배치 크기보다 적으면 **마지막 ID를 복제해서 패딩**. 항상 고정 크기 쿼리 → PreparedStatement 캐시 효율 상승
- **`DYNAMIC`** (기본): 쿼리 시점에 필요한 수의 ID로 IN 절 구성. 메모리 효율 상승
> **실무 팁**: Hibernate 6부터는 `DYNAMIC`이 기본이라 따로 설정할 필요 거의 없음.
 
### 5.5 배치 크기 결정 가이드라인
 
| 값 | 코멘트 |
|---|---|
| 10~50 | Vlad, 블로그들이 흔히 추천하는 범위 |
| 100~500 | 실무에서 많이 사용 |
| 1000+ | DB 한계 주의 (Oracle 11g 이하 IN 1000개 제한) |
 
> 크게 설정할수록 쿼리 수는 줄지만 IN 절이 길어지며 파서·옵티마이저 부담. **테스트로 sweet spot 찾을 것**.
 
### 5.6 `@BatchSize` vs `default_batch_fetch_size` vs `@Fetch(SUBSELECT)`
 
| 방식 | 범위 | 쿼리 형태 |
|---|---|---|
| `default_batch_fetch_size` | 전역 | `WHERE id IN (?, ?, ...)` |
| `@BatchSize` | 특정 엔티티/컬렉션 | 동일 |
| `@Fetch(FetchMode.SUBSELECT)` | 특정 컬렉션 | `WHERE parent_id IN (원본 쿼리)` 서브쿼리 |
 
`SUBSELECT`는 부모 조회 쿼리를 서브쿼리로 재사용하는 방식. 부모가 1000만건이면 서브쿼리가 무거워질 수 있어서 batch fetch보다 조심스럽게 써야 함.

## 6. 자가 검증 Q&A
 
### Q1. Fetch Join + 페이징 시 MultipleBagFetchException이 아니라 HHH000104가 나는 이유?
 
두 에러는 **다른 층위**에서 발생:
 
| 에러 | 발생 시점 | 원인 |
|---|---|---|
| `MultipleBagFetchException` | **세션팩토리 빌드 / 쿼리 파싱** | bag 2개 이상이 동시 fetch 대상 → "아예 실행 불가" |
| `HHH000104` | **쿼리 실행 런타임** | 컬렉션 1개 + `firstResult`/`maxResults` → "실행은 하지만 메모리에서 페이징" |
 
전자는 **컴파일 타임 방어**에 가깝고, 후자는 **런타임 성능 경고**(Hibernate 6+에서는 예외).
 
### Q2. OneToMany 2개 이상 Fetch Join이 왜 안 되는가?
 
두 레이어의 이유
 
**(1) SQL 레벨 — 카티션 곱**
Post(1) × Comment(N) × Tag(M) = `N × M` 행. 서로 무관한 컬렉션이 엮여서 결과셋이 폭발.
 
**(2) 의미론 레벨 — bag의 중복 구분 불가**
bag(순서 없는 List)은 중복을 허용하기 때문에, Hibernate가 "카티션 곱으로 생긴 가짜 중복"과 "원본의 진짜 중복"을 분리할 기준이 없음. List를 Set으로 바꾸면 중복 제거는 되지만, 카티션 곱이라는 성능 문제는 그대로 → 근본 해결은 **쿼리 분할 or batch fetch**.

## 7. 상황별 선택 가이드 (요약)
 
```
fetch 필요?
├─ No  → 그대로 LAZY
└─ Yes
   ├─ ToOne 연관만? → @EntityGraph or Fetch Join
   ├─ 컬렉션 1개 + 페이징 없음? → Fetch Join
   ├─ 컬렉션 1개 + 페이징?
   │   → ToOne만 EntityGraph + 컬렉션 batch fetch
   ├─ 컬렉션 2개 이상? → default_batch_fetch_size
   └─ 읽기 전용? → DTO projection
```
