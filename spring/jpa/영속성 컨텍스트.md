# TIL: 영속성 컨텍스트

## 영속성 컨텍스트

> **정의**: 엔티티를 영구 저장하는 환경. 애플리케이션과 DB 사이의 논리적 공간으로, 엔티티의 생명주기를 관리한다.
 
### 중요한 사실 3가지
1. **EntityManager**가 영속성 컨텍스트에 접근하는 인터페이스 (Hibernate의 `Session`이 구현체)
2. 기본적으로 **트랜잭션 단위**로 생성·소멸됨
3. 내부는 **`Map<식별자, 엔티티 인스턴스>`** 구조 — 이게 1차 캐시의 본체

---

## 영속성 컨텍스트의 기능

### 1. 1차 캐시 (First-Level Cache)
 
- 영속 상태의 엔티티는 내부 Map에 `@Id` 기준으로 저장됨
- `em.find()` 호출 시 1차 캐시 우선 조회 → 없으면 DB 조회 후 캐시에 저장
- 같은 트랜잭션 내에서 같은 엔티티를 여러 번 조회해도 DB 쿼리는 1번
```java
Member m1 = em.find(Member.class, 1L);  // SELECT 쿼리 1회
Member m2 = em.find(Member.class, 1L);  // 쿼리 없음, 캐시 반환
```
 
**현실적 한계**: 트랜잭션 단위라서 전역 캐시처럼 쓸 순 없다. 애플리케이션 성능을 극적으로 올려주진 않음. 1차 캐시의 진짜 가치는 "동일성 보장"과 "변경 감지"의 기반이라는 데 있음.
 
---
 
### 2. 동일성 보장 (Identity Guarantee)
 
- 같은 트랜잭션 내에서 같은 식별자로 조회하면 항상 같은 인스턴스
- Java의 `==` 비교도 `true`
```java
Member a = em.find(Member.class, 1L);
Member b = em.find(Member.class, 1L);
System.out.println(a == b);  // true
```
 
- 기존 MyBatis·JDBC는 매번 새 객체 → `false`
- Identity Map 패턴의 구현. Repeatable Read 수준의 객체 일관성을 애플리케이션 레벨에서 제공
---
 
### 3. 쓰기 지연 (Transactional Write-Behind)
 
- `em.persist()`를 호출해도 INSERT SQL이 즉시 나가지 않음
- 내부의 쓰기 지연 SQL 저장소(ActionQueue)에 보관 → Flush 시점에 일괄 전송
```java
transaction.begin();
 
em.persist(memberA);   // 1차 캐시 저장 + INSERT SQL은 저장소로
em.persist(memberB);
em.persist(memberC);
 
transaction.commit();  // 여기서 flush → 3개 INSERT 한 번에 전송
```
 
**이점**
- JDBC Batch 최적화 가능 (`hibernate.jdbc.batch_size`)
- 네트워크 왕복 감소
- 트랜잭션 원자성 보장
**함정**: `IDENTITY` 전략(MySQL `AUTO_INCREMENT`)은 ID를 알아야 영속성 컨텍스트에 저장 가능 → `persist()` 시점에 즉시 INSERT 발생 → 쓰기 지연 사실상 무효화.
 
---
 
### 4. 변경 감지 (Dirty Checking)
 
- 영속 상태의 엔티티 값을 바꾸기만 하면 자동으로 UPDATE 쿼리 생성
- `em.update()` 같은 메서드 없음 (Hibernate엔 있지만 JPA 표준엔 없음)
```java
@Transactional
public void updateMember(Long id) {
    Member m = em.find(Member.class, id);  // 영속 + 스냅샷 생성
    m.setName("변경된 이름");                // setter만 호출
}   // 커밋 시 flush → 스냅샷과 비교 → 변경분 UPDATE 자동 실행
```
 
**동작 원리**
1. 엔티티가 영속 상태가 될 때 스냅샷(deep copy) 저장
2. Flush 시점에 현재 상태와 스냅샷 비교
3. 차이가 있으면 UPDATE SQL 자동 생성
**매우 중요한 함정**
- 영속 상태에만 적용됨. 준영속·비영속 엔티티는 아무리 바꿔도 무시
- `@Transactional(readOnly = true)` 쓰면 스냅샷을 안 만듦 → 변경 감지 안 됨 (성능 최적화 목적이므로 의도된 동작)
- 기본 전략은 모든 필드 UPDATE. 바뀐 것만 UPDATE 하려면 `@DynamicUpdate`

## 엔티티 생명주기 4가지 상태
 
```
                 new                  persist()              remove()
[비영속 Transient] ────────→ [영속 Managed] ──────────→ [삭제 Removed]
                                │    ↑
               detach/clear/    │    │  merge() / persist()
               close            │    │
                                ↓    │
                         [준영속 Detached]
```
 
### Transient (비영속)
- `new`로 만들었지만 아직 영속성 컨텍스트와 무관
- 순수 Java 객체
### Managed (영속)
- 영속성 컨텍스트가 관리 중
- 1차 캐시, 동일성 보장, 쓰기 지연, 변경 감지 모두 작동
- 진입 경로: `persist()`, `find()`, JPQL 조회, `merge()` 후 반환값
### Detached (준영속)
- 한 번 관리됐다가 분리된 상태
- DB에는 있지만 변경 감지 작동 안 함
- 진입 경로: `detach()`, `clear()`, `close()`, 트랜잭션 종료, 직렬화
### Removed (삭제)
- 삭제 예약 상태. 실제 DELETE는 다음 flush 때
- `em.remove()`

## @Transactional과 영속성 컨텍스트 생명주기
 
### 기본 규칙
> 트랜잭션 시작 = 영속성 컨텍스트 생성
> 트랜잭션 종료 = flush + 영속성 컨텍스트 close
 
```java
@Transactional
public void service() {
    // [PC 시작]
    Member m = repo.findById(1L).get();  // 영속
    m.setName("변경");                    // 변경 감지 예약
    // [flush → PC 종료]
}
// 메서드 종료 후 m은 준영속 상태
```
 
### 같은 트랜잭션 = 같은 PC
- 트랜잭션 전파로 같은 트랜잭션에 참여하는 서비스들은 영속성 컨텍스트 공유
- `REQUIRES_NEW`로 새 트랜잭션을 열면 별도의 영속성 컨텍스트
### @Transactional 없이 Repository 호출 시 — 흔한 함정
```java
// @Transactional 없음
public void bad() {
    Member m = repo.findById(1L).get();  // 내부 TX 열렸다 닫힘 → m은 준영속
    m.setName("변경");                    // 변경 감지 X → DB 반영 안 됨
}
```
서비스 계층에 `@Transactional` 붙이는 이유가 여기에 있다.
 
### OSIV (Open Session In View)
- Spring Boot 기본값: `spring.jpa.open-in-view=true`
- HTTP 요청 시작 ~ 응답 끝까지 영속성 컨텍스트 유지 (+ DB 커넥션도)
- **장점**: 뷰 렌더링 중에도 지연 로딩 가능
- **단점**: DB 커넥션을 오래 붙잡음 → 커넥션 풀 고갈 위험
- **실무 권장**: `false` + OSIV Off 패턴으로 서비스 계층에서 fetch 완료

## Flush 시점
 
> Flush = 영속성 컨텍스트의 변경 내용을 DB에 동기화
> Flush ≠ Commit — flush는 SQL을 보낼 뿐, 커밋은 별개
 
### Flush가 하는 일
1. 변경 감지: dirty 엔티티 검출 → UPDATE SQL 생성
2. 쓰기 지연 저장소의 SQL을 DB로 전송
3. (커밋은 별도 작업)
### Flush 발생 3가지 시점 (FlushModeType.AUTO 기준 = 기본값)
 
| # | 시점 | 설명 |
|---|------|------|
| 1 | 트랜잭션 커밋 직전 | 항상 자동 발생 |
| 2 | JPQL/Criteria 쿼리 실행 전 | 변경사항이 쿼리 결과에 반영되도록 |
| 3 | `em.flush()` 직접 호출 | 수동 동기화 |
 
### JPQL 전에 flush되는 이유
```java
em.persist(memberA);  // 쓰기 지연 저장소에 INSERT 보관
em.persist(memberB);
em.persist(memberC);
 
// flush가 없으면 DB엔 아직 없으므로 쿼리 결과에 안 나옴!
List<Member> result = em.createQuery("select m from Member m", Member.class)
                         .getResultList();
```
Hibernate가 JPQL의 대상 테이블을 분석해서 해당 테이블과 연관된 dirty 엔티티가 있으면 쿼리 실행 전 자동 flush.
 
### FlushModeType 비교
| 모드 | 동작 |
|------|------|
| `AUTO` (기본) | 커밋 + 쿼리 실행 전 flush |
| `COMMIT` | 커밋 시에만 flush (조회 많을 때 성능 최적화용) |

## @Transactional(readOnly = true) 동작 원리
 
앞선 섹션에서 "스냅샷을 안 만들고 자동 flush를 끈다" 정도로만 짚었는데, 실제로는 **Spring / Hibernate / JDBC 세 계층에 걸쳐 서로 다른 최적화**가 일어난다. 이 구조를 이해해야 언제 효과가 있고 언제 효과가 없는지 설명할 수 있다.
 
### 세 계층에서 일어나는 일
 
```
[Spring]     TransactionSynchronizationManager에 readOnly 플래그 저장
   ↓
[Hibernate]  FlushMode.MANUAL 설정 + Session.setDefaultReadOnly(true)
   ↓
[JDBC/DB]    Connection.setReadOnly(true) 호출
```
 
각 계층은 상위 계층이 남긴 정보를 읽어서 자기만의 최적화를 적용한다. Spring 계층 자체로는 아무 것도 안 하고, 그 플래그를 하위 계층이 어떻게 쓰느냐가 본질.
 
### Spring 계층 — 메타데이터 저장이 전부
 
```java
// AbstractPlatformTransactionManager 내부
TransactionSynchronizationManager.setCurrentTransactionReadOnly(true);
```
 
이 값은 나중에 Hibernate Dialect, JDBC 유틸, 그리고 라우팅 DataSource가 참조한다. `@Transactional` 어노테이션 주석에도 명시되어 있듯 **"힌트일 뿐, 쓰기 시도를 강제로 실패시키지 않는다"**. 실제 쓰기 차단은 하위 계층에서 결정된다.
 
### Hibernate 계층 — 가장 극적인 최적화
 
여기가 진짜 성능에 영향을 주는 부분. Spring 5.1부터 두 가지가 동시에 설정된다.
 
**(1) FlushMode.MANUAL 설정**
- 자동 flush 완전 비활성화
- 트랜잭션 커밋 시에도 flush 안 함
- JPQL 실행 전에도 flush 안 함
- `em.flush()`를 직접 호출하지 않는 한 쿼리가 나가지 않음
조회만 하는 트랜잭션에선 flush할 것도 없으니 이 비용 자체가 사라진다.
 
**(2) Session.setDefaultReadOnly(true)** — 이게 핵심이다
 
이 세션에서 로드되는 모든 엔티티를 read-only 모드로 로드한다. 구체적으로는:
- **스냅샷(hydrated state)을 아예 만들지 않음** → 메모리 대략 절반 절감
- 스냅샷이 없으므로 dirty check도 원천 차단
- 버전리스 낙관적 락도 동작 불가 (스냅샷 기반 WHERE 절을 못 만듦)
이 최적화는 Vlad Mihalcea(Hibernate 커미터)가 2018년 Spring에 기여한 [SPR-16956](https://github.com/spring-projects/spring-framework/issues/21494)의 결과다. 그 전까진 Spring이 FlushMode만 바꾸고 스냅샷은 그대로 만들어서 "반쪽짜리 최적화"였다.
 
| 항목 | Spring 5.0 이하 | Spring 5.1+ |
|------|----------------|-------------|
| FlushMode | MANUAL 설정 | MANUAL 설정 |
| 스냅샷 생성 | 생성함 | **생성 안 함** |
| 메모리 사용 | 일반 모드와 동일 | 약 50% 절감 |
 
### JDBC 계층 — DB별로 동작이 갈린다
 
Spring은 JDBC Connection에도 힌트를 전달한다.
 
```java
// DataSourceUtils.prepareConnectionForTransaction 내부
con.setReadOnly(true);
```
 
문제는 이 호출의 의미가 DB마다 다르다는 것.
 
| DB | 동작 |
|----|------|
| MySQL (InnoDB) | 힌트로 처리. 쓰기 시도해도 허용됨 |
| PostgreSQL | `SET TRANSACTION READ ONLY` 전송. 쓰기 시도 시 실제로 에러 |
| Oracle | Read-Only Transaction 기능 사용. 트랜잭션 시작 시점 스냅샷으로 일관성 보장 |
 
"readOnly는 그냥 힌트일 뿐"이라는 말은 반은 맞고 반은 틀리다. PostgreSQL처럼 실제로 쓰기를 막는 DB도 있다.
 
### Master/Slave Replication 환경에서의 확장 포인트
 
`readOnly=true`의 진짜 위력은 Replication 아키텍처에서 나온다.
 
```java
public class ReplicationRoutingDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
                ? "read"    // Read Replica로 라우팅
                : "write";  // Master로 라우팅
    }
}
```
 
`LazyConnectionDataSourceProxy`로 감싸서 커넥션 획득을 실제 SQL 실행 직전까지 지연시켜야 라우팅이 올바르게 동작한다 (그러지 않으면 트랜잭션 속성이 확정되기 전에 커넥션이 잡혀버림).
 
### Spring Data JPA Repository의 숨은 기본값
 
많이 놓치는 사실: Spring Data JPA의 `SimpleJpaRepository`는 **이미 클래스 레벨에 `@Transactional(readOnly = true)`가 붙어 있다**.
 
```java
@Repository
@Transactional(readOnly = true)  // 클래스 레벨 기본값
public class SimpleJpaRepository<T, ID> implements JpaRepository<T, ID> {
 
    @Transactional  // 쓰기 메서드만 오버라이드
    public <S extends T> S save(S entity) { ... }
}
```
 
즉 서비스에 `@Transactional` 없이 `repository.findById()`만 호출해도 자동으로 read-only 트랜잭션이 열린다. 다만 **여러 Repository를 묶거나 지연 로딩을 쓰려면 서비스 계층에 `@Transactional(readOnly=true)`가 필수**다 — 그래야 하나의 트랜잭션으로 묶이고 영속성 컨텍스트가 공유된다.

### 함정 모음
 
**함정 1. readOnly=true에서 엔티티 수정해도 조용히 무시된다**
```java
@Transactional(readOnly = true)
public void silentBug() {
    Review r = reviewRepository.findById(1L).get();
    r.setTitle("변경");  // 에러 안 남. DB 반영도 안 됨. 로그도 없음.
}
```
디버깅이 어려워지므로 수정 메서드엔 반드시 `@Transactional`(기본값 false) 쓰기.
 
**함정 2. 외부 트랜잭션이 readOnly=false면 내부의 readOnly=true는 무시된다**
```java
@Transactional  // readOnly=false
public void outer() {
    inner();  // inner의 readOnly=true가 적용되지 않음
}
 
@Transactional(readOnly = true)
public void inner() { ... }
```
`Propagation.REQUIRED`는 이미 열린 트랜잭션에 참여할 뿐 속성을 갱신하지 않는다. 분리가 꼭 필요하면 `REQUIRES_NEW`를 써야 하지만 별도 트랜잭션이 되므로 신중히.
 
**함정 3. readOnly=true에서 save() 호출 시 DB마다 결과가 다르다**
- MySQL: 쓰기 성공 (경고만)
- PostgreSQL: 에러 발생
- DB 포팅 시 깨질 수 있음. 애초에 이런 코드를 쓰지 말 것.

**함정 4. JPA가 아닌 JDBC/MyBatis에선 Hibernate 최적화가 없다**
스냅샷·dirty check는 Hibernate 전용 개념. JDBC/MyBatis에선 `Connection.setReadOnly(true)` 호출과 Replica 라우팅 효과만 남는다.
