# HTTP 완벽가이드 1장 — HTTP 개관

## 웹에서 일어나는 일의 전체 그림

HTTP는 클라이언트(브라우저)와 서버 사이에서 데이터를 주고받는 프로토콜이다. 웹에서 보이는 텍스트, 이미지, 영상 전부 HTTP를 통해 옮겨진다.

---

## 리소스

웹 서버가 관리하는 콘텐츠를 통틀어 리소스라고 부른다. 정적 파일(HTML, 이미지)만 리소스가 아니라, 요청에 따라 동적으로 생성되는 콘텐츠도 리소스다.

### MIME 타입

HTTP는 전송되는 데이터에 MIME 타입이라는 라벨을 붙인다. `주 타입/부 타입` 형식이다.

- `text/html` — HTML 문서
- `text/plain` — 일반 텍스트
- `image/jpeg` — JPEG 이미지
- `application/json` — JSON 데이터

브라우저는 이 라벨을 보고 데이터를 어떻게 처리할지 결정한다.

### URI, URL, URN

- **URI** — 리소스를 식별하는 통합 개념
- **URL** — 리소스의 **위치**로 식별 (실무에서 거의 이걸 쓴다)
- **URN** — 리소스의 **이름**으로 식별 (아직 널리 쓰이지 않음)

URI = URL + URN이라고 보면 된다. 대부분의 경우 URI와 URL은 같은 의미로 쓰인다.

---

## 트랜잭션

HTTP 통신의 단위. 요청(request) + 응답(response) 한 쌍이 하나의 트랜잭션이다.

### HTTP 메서드

서버에게 어떤 동작을 해달라고 알려주는 것.

| 메서드 | 의미 |
|--------|------|
| GET | 리소스를 달라 |
| PUT | 이 데이터를 저장해라 |
| DELETE | 리소스를 삭제해라 |
| POST | 이 데이터를 처리해라 |
| HEAD | 헤더만 보내줘 (본문 필요 없음) |

### 상태 코드

응답에 붙는 세 자리 숫자. 요청이 어떻게 처리됐는지 알려준다.

| 코드 | 의미 |
|------|------|
| 200 | 정상 처리 |
| 302 | 리다이렉트 (다른 URL로 가라) |
| 404 | 리소스 없음 |
| 500 | 서버 에러 |

하나의 웹 페이지를 보여주기 위해 여러 트랜잭션이 발생한다. HTML 하나 받고, 그 안의 이미지·CSS·JS 각각에 대해 추가 트랜잭션이 일어난다.

---

## 메시지

HTTP 메시지는 단순한 줄 단위 텍스트 구조다. 사람이 읽을 수 있다.

```
시작줄        → 무엇을 할지 / 무엇이 일어났는지
헤더          → key: value 형태의 메타데이터 (0개 이상)
(빈 줄)
본문          → 실제 데이터 (없을 수도 있음)
```

요청 메시지 예시:
```
GET /index.html HTTP/1.1
Host: www.example.com
Accept: text/html
```

응답 메시지 예시:
```
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 617

<html>...</html>
```

---

## TCP/IP

HTTP는 TCP 위에서 동작한다. 데이터가 손실되거나 순서가 뒤바뀌지 않게 TCP가 보장해준다.

통신 순서:

1. 클라이언트가 URL에서 호스트명 추출
2. DNS로 호스트명 → IP 주소 변환
3. 포트 번호 확인 (없으면 80)
4. 해당 IP:포트로 TCP 커넥션 수립
5. HTTP 요청 전송
6. HTTP 응답 수신
7. 커넥션 종료

---

## 웹의 구성요소

HTTP가 동작하는 환경에는 클라이언트·서버 외에도 여러 구성요소가 있다.

- **프록시** — 클라이언트와 서버 사이에 위치. 요청을 중계하면서 필터링, 로깅, 캐싱 등을 수행한다.
- **캐시** — 자주 요청되는 리소스의 사본을 저장해서 응답 속도를 높이고 서버 부하를 줄인다. 프록시 캐시가 대표적.
- **게이트웨이** — 다른 프로토콜이나 애플리케이션과 HTTP를 연결하는 중간 서버. HTTP → DB 같은 변환을 담당한다.
- **터널** — HTTP 커넥션 위에 다른 프로토콜 트래픽을 그대로 흘려보내는 통로. 대표적으로 SSL 터널링.
- **에이전트** — HTTP 요청을 보내는 클라이언트 소프트웨어. 브라우저가 대표적이지만 웹 크롤러 같은 자동화 에이전트도 있다.

---

## 핵심 한 줄

> HTTP는 요청-응답 기반의 텍스트 프로토콜이고, TCP 위에서 동작하며, 웹의 모든 데이터 전송을 담당한다.

# HTTP 완벽가이드 2장 — URL과 리소스

## URL이 하는 일

URL은 인터넷 리소스의 주소다. URL 덕분에 프로토콜이 뭐든 간에 하나의 통일된 형식으로 리소스에 접근할 수 있게 됐다. URL이 없던 시절에는 애플리케이션마다 접근 방식이 달랐다.

---

## URL 문법

기본 구조:

```
<스킴>://<사용자>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>?<질의>#<프래그먼트>
```

전부 다 쓰는 경우는 거의 없다. 스킴·호스트·경로가 핵심이고 나머지는 필요할 때 붙는다.

### 스킴 (Scheme)

리소스에 접근할 프로토콜을 지정한다. 대소문자 구분 안 함.

- `http` — 기본 포트 80
- `https` — 기본 포트 443, TLS 암호화
- `ftp` — 파일 전송
- `mailto` — 이메일 주소 지정

### 호스트 + 포트

리소스가 있는 서버의 위치. 호스트는 도메인명이나 IP 주소로 쓴다. 포트를 생략하면 스킴의 기본 포트가 적용된다.

```
http://www.example.com:8080/index.html
         ───────────── ────
           호스트       포트
```

### 경로 (Path)

서버 내 리소스의 위치를 `/`로 구분한 계층 구조로 표현한다. 유닉스 파일 시스템 경로와 비슷하다.

```
http://www.example.com/docs/guide/ch01.html
                       ────────────────────
                              경로
```

### 파라미터 (Parameters)

경로 조각에 `;`으로 추가 정보를 붙인다. 서버가 리소스에 접근하는 데 필요한 부가 입력값이다.

```
ftp://prep.ai.mit.edu/pub/gnu;type=d
```

실무에서 자주 보이진 않지만, FTP 등에서 간혹 쓰인다.

### 질의 문자열 (Query String)

`?` 뒤에 `key=value` 쌍을 `&`로 연결한다. 가장 익숙한 형태.

```
http://www.example.com/search?q=http&lang=ko
```

질의 문자열의 포맷에 대한 공식 표준은 없다. 관례적으로 `key=value&key=value` 형태를 쓸 뿐이다.

### 프래그먼트 (Fragment)

`#` 뒤에 오는 값. 리소스의 특정 부분(HTML 문서의 특정 섹션 등)을 가리킨다.

```
http://www.example.com/docs/guide.html#chapter3
```

**중요: 서버에 전송되지 않는다.** 브라우저가 서버에서 리소스를 통째로 받은 뒤, 프래그먼트를 보고 해당 부분으로 스크롤하는 식으로 처리한다. 서버는 프래그먼트의 존재를 모른다.

---

## 단축 URL

### 상대 URL

매번 절대 URL을 쓰면 길고 번거롭다. 기저(base) URL을 기준으로 상대 경로만 쓸 수 있다.

```
기저 URL:  http://www.example.com/docs/index.html
상대 URL:  ./guide.html
결과:      http://www.example.com/docs/guide.html
```

base URL 결정 순서:
1. `<base>` 태그에 명시된 URL
2. 현재 리소스의 URL
3. 불완전한 URL (이 경우 해석이 실패할 수 있음)

상대 URL → 절대 URL 변환은 RFC 2396에 정의된 알고리즘을 따른다. 스킴·호스트·경로를 순서대로 상속하면서 빈 부분을 채워나가는 방식이다.

### URL 확장

브라우저가 제공하는 편의 기능이지 HTTP 스펙은 아니다.

- **호스트명 확장** — `yahoo`만 치면 `www.yahoo.com`으로 자동 완성
- **히스토리 확장** — 이전 방문 기록을 기반으로 자동 완성

프록시를 사용하는 환경에서는 호스트명 확장이 문제를 일으킬 수 있다.

---

## URL 인코딩

URL에 아무 문자나 넣을 수 없다. 안전하지 않은 문자는 **퍼센트 인코딩**(`%` + 16진수 두 자리)으로 이스케이프한다.

```
공백 → %20
~ → %7E
% → %25
```

### 문자 제한

URL에 쓸 수 있는 문자는 US-ASCII의 일부로 제한된다.

- **안전한 문자** — 알파벳, 숫자, 몇몇 특수문자(`-`, `_`, `.`, `~`)는 인코딩 없이 그대로 쓸 수 있다.
- **예약 문자** — `/`, `?`, `#`, `&`, `=` 등은 URL 구조에서 특별한 의미가 있다. 이걸 데이터로 쓰려면 반드시 인코딩해야 한다.
- **그 외** — 비영어 문자, 바이너리 데이터 등은 전부 인코딩 대상이다.

원칙은 단순하다. **확실히 안전한 문자가 아니면 인코딩하는 게 맞다.** 클라이언트 애플리케이션에서 URL을 조립할 때 이 규칙을 지켜야 한다. 전송 전에 인코딩하지 않으면 서버 쪽에서 해석이 꼬일 수 있다.

---

## URL의 한계 — URN의 등장

URL은 리소스의 **위치**를 가리키기 때문에 근본적인 한계가 있다. 리소스가 이동하면 URL이 깨진다. 링크가 죽는 것이다.

이 문제를 해결하기 위해 **URN**(Uniform Resource Name)이 제안됐다. 위치가 아니라 리소스 고유의 이름을 부여하는 방식이다.

```
urn:isbn:0451450523
```

리소스가 어디로 옮겨가든 이름은 변하지 않으니 링크가 안 깨진다는 발상이다. 하지만 URN이 동작하려면 이름을 실제 위치로 매핑하는 인프라가 필요한데, 이 전환 작업이 방대해서 아직까지 URL이 지배적이다.

---

## 핵심 정리

> URL은 `스킴://호스트/경로`가 핵심 구조고, 안전하지 않은 문자는 퍼센트 인코딩으로 처리한다. 프래그먼트는 서버에 전송되지 않는다. URL은 위치 기반이라 리소스가 이동하면 깨지는 한계가 있다.

# HTTP 완벽가이드 3장 — HTTP 메시지

## 메시지의 흐름

HTTP 메시지는 항상 **다운스트림**으로 흐른다. 클라이언트든 서버든 프록시든, 보내는 쪽이 상류(upstream)이고 받는 쪽이 하류(downstream)이다. 요청이든 응답이든 방향과 무관하게 메시지는 항상 다운스트림으로 간다.

---

## 메시지의 구조

모든 HTTP 메시지는 세 부분으로 이루어진다.

```
시작줄 (start line)       ← 한 줄. 무엇을 해야 하는지 / 무슨 일이 일어났는지
헤더 (headers)            ← 0개 이상의 key: value 쌍
                          ← 빈 줄 (CRLF) — 헤더의 끝을 알림
본문 (body)               ← 있을 수도, 없을 수도 있음
```

시작줄과 헤더는 줄 단위 텍스트(ASCII)이고, 본문은 텍스트든 바이너리든 아무 데이터나 담을 수 있다.

### 요청 메시지의 시작줄 (요청줄)

```
GET /docs/index.html HTTP/1.1
─── ──────────────── ────────
메서드    URL          버전
```

### 응답 메시지의 시작줄 (응답줄)

```
HTTP/1.1 200 OK
──────── ─── ──
 버전   코드 사유구절
```

사유 구절(reason phrase)은 사람이 읽기 위한 것이고, 프로그램은 상태 코드만 보면 된다.

---

## 메서드

| 메서드 | 설명 | 본문 |
|--------|------|------|
| GET | 리소스 조회 | 없음 |
| HEAD | GET과 같지만 헤더만 반환 | 없음 |
| POST | 서버에 데이터 전송 | 있음 |
| PUT | 리소스 전체 교체 (없으면 생성) | 있음 |
| PATCH | 리소스 부분 수정 | 있음 |
| DELETE | 리소스 삭제 | 없음 |
| TRACE | 메시지가 프록시를 거치는 경로 추적 | 없음 |
| OPTIONS | 서버가 지원하는 메서드 확인 | 없음 |

### 안전한 메서드

GET, HEAD는 서버 상태를 변경하지 않으므로 "안전한 메서드"라고 부른다. 다만 이건 HTTP 스펙이 보장하는 게 아니라 서버 구현자가 지켜야 할 규약이다. GET으로 DB를 변경하는 서버를 만들 수도 있지만, 그건 스펙 위반이다.

### HEAD

GET과 동일하게 동작하되 본문 없이 헤더만 돌려준다. 리소스의 존재 여부, 타입, 변경 시간 등을 확인할 때 유용하다. 서버는 HEAD 응답의 헤더가 GET 응답과 정확히 동일해야 한다.

### TRACE

요청 메시지가 프록시 체인을 거치면서 어떻게 변경되는지 확인하는 진단용 메서드. 서버는 받은 요청 메시지를 그대로 본문에 담아 돌려준다. 보안 취약점 때문에 대부분의 서버에서 비활성화한다.

---

## 상태 코드

### 1xx — 정보성

| 코드 | 의미 |
|------|------|
| 100 Continue | 요청의 나머지를 보내도 된다 |
| 101 Switching Protocols | 프로토콜 전환 승인 (ex: WebSocket 업그레이드) |

`100 Continue`는 클라이언트가 큰 본문을 보내기 전에, 서버가 받아줄 의향이 있는지 먼저 확인하는 용도다. 요청 헤더에 `Expect: 100-continue`를 넣으면 서버가 100 또는 에러 코드로 응답한다.

### 2xx — 성공

| 코드 | 의미 |
|------|------|
| 200 OK | 정상 처리 |
| 201 Created | 리소스 생성 완료 |
| 202 Accepted | 요청 접수됨, 아직 처리 완료는 아님 |
| 204 No Content | 성공했지만 본문 없음 |

### 3xx — 리다이렉션

| 코드 | 의미 |
|------|------|
| 301 Moved Permanently | 영구 이동. 북마크도 갱신해야 함 |
| 302 Found | 임시 이동. 원래 URL을 계속 써야 함 |
| 304 Not Modified | 캐시된 버전을 써도 된다 |
| 307 Temporary Redirect | 302와 비슷하지만 메서드를 변경하면 안 됨 |

`304`는 조건부 요청(`If-Modified-Since`, `If-None-Match`)에 대한 응답이다. 본문을 보내지 않아도 되니 대역폭을 아낄 수 있다.

### 4xx — 클라이언트 에러

| 코드 | 의미 |
|------|------|
| 400 Bad Request | 잘못된 요청 |
| 401 Unauthorized | 인증 필요 |
| 403 Forbidden | 권한 없음 (인증과 무관) |
| 404 Not Found | 리소스 없음 |
| 405 Method Not Allowed | 해당 메서드를 지원하지 않음 |

### 5xx — 서버 에러

| 코드 | 의미 |
|------|------|
| 500 Internal Server Error | 서버 내부 에러 |
| 502 Bad Gateway | 게이트웨이/프록시가 업스트림에서 잘못된 응답을 받음 |
| 503 Service Unavailable | 서버 일시적 과부하 또는 점검 중 |
| 504 Gateway Timeout | 게이트웨이/프록시가 업스트림 응답을 제시간에 못 받음 |

## 헤더

메시지에 대한 부가 정보를 `이름: 값` 형태로 전달한다. 여러 줄에 걸쳐 올 수 있고(줄 접기), 순서는 상관없다.

### 일반 헤더 (요청·응답 모두 사용)

- `Connection` — 커넥션 관리 (`keep-alive`, `close`)
- `Date` — 메시지 생성 일시
- `Transfer-Encoding` — 본문 전송 인코딩 방식 (`chunked` 등)
- `Cache-Control` — 캐시 지시자

### 요청 헤더

- `Host` — 요청 대상 서버의 호스트명 (HTTP/1.1 필수)
- `User-Agent` — 클라이언트 애플리케이션 정보
- `Accept` — 클라이언트가 처리 가능한 미디어 타입
- `Accept-Encoding` — 처리 가능한 인코딩 (`gzip`, `deflate`)
- `If-Modified-Since` — 조건부 요청, 변경된 경우에만 응답
- `Authorization` — 인증 정보

### 응답 헤더

- `Server` — 서버 소프트웨어 정보
- `Set-Cookie` — 클라이언트에 쿠키 설정
- `WWW-Authenticate` — 401 응답 시 인증 방법 안내

### 엔터티 헤더 (본문에 대한 정보)

- `Content-Type` — 본문의 미디어 타입
- `Content-Length` — 본문 크기 (바이트)
- `Content-Encoding` — 본문 압축 방식
- `Last-Modified` — 리소스 마지막 수정 시각
- `ETag` — 리소스의 버전 식별자

### 확장 헤더

표준에 정의되지 않은 커스텀 헤더도 자유롭게 추가할 수 있다. 수신 측이 모르는 헤더는 무시하면 된다.

## 핵심 정리

> HTTP 메시지는 시작줄 + 헤더 + 본문의 단순한 텍스트 구조다. 요청줄에는 메서드·URL·버전이, 응답줄에는 버전·상태코드·사유구절이 온다. 메서드는 서버에게 원하는 동작을 알려주고, 상태 코드는 처리 결과를 알려준다. 헤더로 부가 정보를 전달한다.
