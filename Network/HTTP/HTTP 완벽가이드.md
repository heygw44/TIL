# HTTP 완벽가이드 1장 — HTTP 개관

## 웹에서 일어나는 일의 전체 그림

HTTP는 클라이언트(브라우저)와 서버 사이에서 데이터를 주고받는 프로토콜이다. 웹에서 보이는 텍스트, 이미지, 영상 전부 HTTP를 통해 옮겨진다.

---

## 리소스

웹 서버가 관리하는 콘텐츠를 통틀어 리소스라고 부른다. 정적 파일(HTML, 이미지)만 리소스가 아니라, 요청에 따라 동적으로 생성되는 콘텐츠도 리소스다.

### MIME 타입

HTTP는 전송되는 데이터에 MIME 타입이라는 라벨을 붙인다. `주 타입/부 타입` 형식이다.

- `text/html` — HTML 문서
- `text/plain` — 일반 텍스트
- `image/jpeg` — JPEG 이미지
- `application/json` — JSON 데이터

브라우저는 이 라벨을 보고 데이터를 어떻게 처리할지 결정한다.

### URI, URL, URN

- **URI** — 리소스를 식별하는 통합 개념
- **URL** — 리소스의 **위치**로 식별 (실무에서 거의 이걸 쓴다)
- **URN** — 리소스의 **이름**으로 식별 (아직 널리 쓰이지 않음)

URI = URL + URN이라고 보면 된다. 대부분의 경우 URI와 URL은 같은 의미로 쓰인다.

---

## 트랜잭션

HTTP 통신의 단위. 요청(request) + 응답(response) 한 쌍이 하나의 트랜잭션이다.

### HTTP 메서드

서버에게 어떤 동작을 해달라고 알려주는 것.

| 메서드 | 의미 |
|--------|------|
| GET | 리소스를 달라 |
| PUT | 이 데이터를 저장해라 |
| DELETE | 리소스를 삭제해라 |
| POST | 이 데이터를 처리해라 |
| HEAD | 헤더만 보내줘 (본문 필요 없음) |

### 상태 코드

응답에 붙는 세 자리 숫자. 요청이 어떻게 처리됐는지 알려준다.

| 코드 | 의미 |
|------|------|
| 200 | 정상 처리 |
| 302 | 리다이렉트 (다른 URL로 가라) |
| 404 | 리소스 없음 |
| 500 | 서버 에러 |

하나의 웹 페이지를 보여주기 위해 여러 트랜잭션이 발생한다. HTML 하나 받고, 그 안의 이미지·CSS·JS 각각에 대해 추가 트랜잭션이 일어난다.

---

## 메시지

HTTP 메시지는 단순한 줄 단위 텍스트 구조다. 사람이 읽을 수 있다.

```
시작줄        → 무엇을 할지 / 무엇이 일어났는지
헤더          → key: value 형태의 메타데이터 (0개 이상)
(빈 줄)
본문          → 실제 데이터 (없을 수도 있음)
```

요청 메시지 예시:
```
GET /index.html HTTP/1.1
Host: www.example.com
Accept: text/html
```

응답 메시지 예시:
```
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 617

<html>...</html>
```

---

## TCP/IP

HTTP는 TCP 위에서 동작한다. 데이터가 손실되거나 순서가 뒤바뀌지 않게 TCP가 보장해준다.

통신 순서:

1. 클라이언트가 URL에서 호스트명 추출
2. DNS로 호스트명 → IP 주소 변환
3. 포트 번호 확인 (없으면 80)
4. 해당 IP:포트로 TCP 커넥션 수립
5. HTTP 요청 전송
6. HTTP 응답 수신
7. 커넥션 종료

---

## 웹의 구성요소

HTTP가 동작하는 환경에는 클라이언트·서버 외에도 여러 구성요소가 있다.

- **프록시** — 클라이언트와 서버 사이에 위치. 요청을 중계하면서 필터링, 로깅, 캐싱 등을 수행한다.
- **캐시** — 자주 요청되는 리소스의 사본을 저장해서 응답 속도를 높이고 서버 부하를 줄인다. 프록시 캐시가 대표적.
- **게이트웨이** — 다른 프로토콜이나 애플리케이션과 HTTP를 연결하는 중간 서버. HTTP → DB 같은 변환을 담당한다.
- **터널** — HTTP 커넥션 위에 다른 프로토콜 트래픽을 그대로 흘려보내는 통로. 대표적으로 SSL 터널링.
- **에이전트** — HTTP 요청을 보내는 클라이언트 소프트웨어. 브라우저가 대표적이지만 웹 크롤러 같은 자동화 에이전트도 있다.

---

## 핵심 한 줄

> HTTP는 요청-응답 기반의 텍스트 프로토콜이고, TCP 위에서 동작하며, 웹의 모든 데이터 전송을 담당한다.

# HTTP 완벽가이드 2장 — URL과 리소스

## URL이 하는 일

URL은 인터넷 리소스의 주소다. URL 덕분에 프로토콜이 뭐든 간에 하나의 통일된 형식으로 리소스에 접근할 수 있게 됐다. URL이 없던 시절에는 애플리케이션마다 접근 방식이 달랐다.

---

## URL 문법

기본 구조:

```
<스킴>://<사용자>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>?<질의>#<프래그먼트>
```

전부 다 쓰는 경우는 거의 없다. 스킴·호스트·경로가 핵심이고 나머지는 필요할 때 붙는다.

### 스킴 (Scheme)

리소스에 접근할 프로토콜을 지정한다. 대소문자 구분 안 함.

- `http` — 기본 포트 80
- `https` — 기본 포트 443, TLS 암호화
- `ftp` — 파일 전송
- `mailto` — 이메일 주소 지정

### 호스트 + 포트

리소스가 있는 서버의 위치. 호스트는 도메인명이나 IP 주소로 쓴다. 포트를 생략하면 스킴의 기본 포트가 적용된다.

```
http://www.example.com:8080/index.html
         ───────────── ────
           호스트       포트
```

### 경로 (Path)

서버 내 리소스의 위치를 `/`로 구분한 계층 구조로 표현한다. 유닉스 파일 시스템 경로와 비슷하다.

```
http://www.example.com/docs/guide/ch01.html
                       ────────────────────
                              경로
```

### 파라미터 (Parameters)

경로 조각에 `;`으로 추가 정보를 붙인다. 서버가 리소스에 접근하는 데 필요한 부가 입력값이다.

```
ftp://prep.ai.mit.edu/pub/gnu;type=d
```

실무에서 자주 보이진 않지만, FTP 등에서 간혹 쓰인다.

### 질의 문자열 (Query String)

`?` 뒤에 `key=value` 쌍을 `&`로 연결한다. 가장 익숙한 형태.

```
http://www.example.com/search?q=http&lang=ko
```

질의 문자열의 포맷에 대한 공식 표준은 없다. 관례적으로 `key=value&key=value` 형태를 쓸 뿐이다.

### 프래그먼트 (Fragment)

`#` 뒤에 오는 값. 리소스의 특정 부분(HTML 문서의 특정 섹션 등)을 가리킨다.

```
http://www.example.com/docs/guide.html#chapter3
```

**중요: 서버에 전송되지 않는다.** 브라우저가 서버에서 리소스를 통째로 받은 뒤, 프래그먼트를 보고 해당 부분으로 스크롤하는 식으로 처리한다. 서버는 프래그먼트의 존재를 모른다.

---

## 단축 URL

### 상대 URL

매번 절대 URL을 쓰면 길고 번거롭다. 기저(base) URL을 기준으로 상대 경로만 쓸 수 있다.

```
기저 URL:  http://www.example.com/docs/index.html
상대 URL:  ./guide.html
결과:      http://www.example.com/docs/guide.html
```

base URL 결정 순서:
1. `<base>` 태그에 명시된 URL
2. 현재 리소스의 URL
3. 불완전한 URL (이 경우 해석이 실패할 수 있음)

상대 URL → 절대 URL 변환은 RFC 2396에 정의된 알고리즘을 따른다. 스킴·호스트·경로를 순서대로 상속하면서 빈 부분을 채워나가는 방식이다.

### URL 확장

브라우저가 제공하는 편의 기능이지 HTTP 스펙은 아니다.

- **호스트명 확장** — `yahoo`만 치면 `www.yahoo.com`으로 자동 완성
- **히스토리 확장** — 이전 방문 기록을 기반으로 자동 완성

프록시를 사용하는 환경에서는 호스트명 확장이 문제를 일으킬 수 있다.

---

## URL 인코딩

URL에 아무 문자나 넣을 수 없다. 안전하지 않은 문자는 **퍼센트 인코딩**(`%` + 16진수 두 자리)으로 이스케이프한다.

```
공백 → %20
~ → %7E
% → %25
```

### 문자 제한

URL에 쓸 수 있는 문자는 US-ASCII의 일부로 제한된다.

- **안전한 문자** — 알파벳, 숫자, 몇몇 특수문자(`-`, `_`, `.`, `~`)는 인코딩 없이 그대로 쓸 수 있다.
- **예약 문자** — `/`, `?`, `#`, `&`, `=` 등은 URL 구조에서 특별한 의미가 있다. 이걸 데이터로 쓰려면 반드시 인코딩해야 한다.
- **그 외** — 비영어 문자, 바이너리 데이터 등은 전부 인코딩 대상이다.

원칙은 단순하다. **확실히 안전한 문자가 아니면 인코딩하는 게 맞다.** 클라이언트 애플리케이션에서 URL을 조립할 때 이 규칙을 지켜야 한다. 전송 전에 인코딩하지 않으면 서버 쪽에서 해석이 꼬일 수 있다.

---

## URL의 한계 — URN의 등장

URL은 리소스의 **위치**를 가리키기 때문에 근본적인 한계가 있다. 리소스가 이동하면 URL이 깨진다. 링크가 죽는 것이다.

이 문제를 해결하기 위해 **URN**(Uniform Resource Name)이 제안됐다. 위치가 아니라 리소스 고유의 이름을 부여하는 방식이다.

```
urn:isbn:0451450523
```

리소스가 어디로 옮겨가든 이름은 변하지 않으니 링크가 안 깨진다는 발상이다. 하지만 URN이 동작하려면 이름을 실제 위치로 매핑하는 인프라가 필요한데, 이 전환 작업이 방대해서 아직까지 URL이 지배적이다.

---

## 핵심 정리

> URL은 `스킴://호스트/경로`가 핵심 구조고, 안전하지 않은 문자는 퍼센트 인코딩으로 처리한다. 프래그먼트는 서버에 전송되지 않는다. URL은 위치 기반이라 리소스가 이동하면 깨지는 한계가 있다.

# HTTP 완벽가이드 3장 — HTTP 메시지

## 메시지의 흐름

HTTP 메시지는 항상 **다운스트림**으로 흐른다. 클라이언트든 서버든 프록시든, 보내는 쪽이 상류(upstream)이고 받는 쪽이 하류(downstream)이다. 요청이든 응답이든 방향과 무관하게 메시지는 항상 다운스트림으로 간다.

---

## 메시지의 구조

모든 HTTP 메시지는 세 부분으로 이루어진다.

```
시작줄 (start line)       ← 한 줄. 무엇을 해야 하는지 / 무슨 일이 일어났는지
헤더 (headers)            ← 0개 이상의 key: value 쌍
                          ← 빈 줄 (CRLF) — 헤더의 끝을 알림
본문 (body)               ← 있을 수도, 없을 수도 있음
```

시작줄과 헤더는 줄 단위 텍스트(ASCII)이고, 본문은 텍스트든 바이너리든 아무 데이터나 담을 수 있다.

### 요청 메시지의 시작줄 (요청줄)

```
GET /docs/index.html HTTP/1.1
─── ──────────────── ────────
메서드    URL          버전
```

### 응답 메시지의 시작줄 (응답줄)

```
HTTP/1.1 200 OK
──────── ─── ──
 버전   코드 사유구절
```

사유 구절(reason phrase)은 사람이 읽기 위한 것이고, 프로그램은 상태 코드만 보면 된다.

---

## 메서드

| 메서드 | 설명 | 본문 |
|--------|------|------|
| GET | 리소스 조회 | 없음 |
| HEAD | GET과 같지만 헤더만 반환 | 없음 |
| POST | 서버에 데이터 전송 | 있음 |
| PUT | 리소스 전체 교체 (없으면 생성) | 있음 |
| PATCH | 리소스 부분 수정 | 있음 |
| DELETE | 리소스 삭제 | 없음 |
| TRACE | 메시지가 프록시를 거치는 경로 추적 | 없음 |
| OPTIONS | 서버가 지원하는 메서드 확인 | 없음 |

### 안전한 메서드

GET, HEAD는 서버 상태를 변경하지 않으므로 "안전한 메서드"라고 부른다. 다만 이건 HTTP 스펙이 보장하는 게 아니라 서버 구현자가 지켜야 할 규약이다. GET으로 DB를 변경하는 서버를 만들 수도 있지만, 그건 스펙 위반이다.

### HEAD

GET과 동일하게 동작하되 본문 없이 헤더만 돌려준다. 리소스의 존재 여부, 타입, 변경 시간 등을 확인할 때 유용하다. 서버는 HEAD 응답의 헤더가 GET 응답과 정확히 동일해야 한다.

### TRACE

요청 메시지가 프록시 체인을 거치면서 어떻게 변경되는지 확인하는 진단용 메서드. 서버는 받은 요청 메시지를 그대로 본문에 담아 돌려준다. 보안 취약점 때문에 대부분의 서버에서 비활성화한다.

---

## 상태 코드

### 1xx — 정보성

| 코드 | 의미 |
|------|------|
| 100 Continue | 요청의 나머지를 보내도 된다 |
| 101 Switching Protocols | 프로토콜 전환 승인 (ex: WebSocket 업그레이드) |

`100 Continue`는 클라이언트가 큰 본문을 보내기 전에, 서버가 받아줄 의향이 있는지 먼저 확인하는 용도다. 요청 헤더에 `Expect: 100-continue`를 넣으면 서버가 100 또는 에러 코드로 응답한다.

### 2xx — 성공

| 코드 | 의미 |
|------|------|
| 200 OK | 정상 처리 |
| 201 Created | 리소스 생성 완료 |
| 202 Accepted | 요청 접수됨, 아직 처리 완료는 아님 |
| 204 No Content | 성공했지만 본문 없음 |

### 3xx — 리다이렉션

| 코드 | 의미 |
|------|------|
| 301 Moved Permanently | 영구 이동. 북마크도 갱신해야 함 |
| 302 Found | 임시 이동. 원래 URL을 계속 써야 함 |
| 304 Not Modified | 캐시된 버전을 써도 된다 |
| 307 Temporary Redirect | 302와 비슷하지만 메서드를 변경하면 안 됨 |

`304`는 조건부 요청(`If-Modified-Since`, `If-None-Match`)에 대한 응답이다. 본문을 보내지 않아도 되니 대역폭을 아낄 수 있다.

### 4xx — 클라이언트 에러

| 코드 | 의미 |
|------|------|
| 400 Bad Request | 잘못된 요청 |
| 401 Unauthorized | 인증 필요 |
| 403 Forbidden | 권한 없음 (인증과 무관) |
| 404 Not Found | 리소스 없음 |
| 405 Method Not Allowed | 해당 메서드를 지원하지 않음 |

### 5xx — 서버 에러

| 코드 | 의미 |
|------|------|
| 500 Internal Server Error | 서버 내부 에러 |
| 502 Bad Gateway | 게이트웨이/프록시가 업스트림에서 잘못된 응답을 받음 |
| 503 Service Unavailable | 서버 일시적 과부하 또는 점검 중 |
| 504 Gateway Timeout | 게이트웨이/프록시가 업스트림 응답을 제시간에 못 받음 |

---

## 헤더

메시지에 대한 부가 정보를 `이름: 값` 형태로 전달한다. 여러 줄에 걸쳐 올 수 있고(줄 접기), 순서는 상관없다.

### 일반 헤더 (요청·응답 모두 사용)

- `Connection` — 커넥션 관리 (`keep-alive`, `close`)
- `Date` — 메시지 생성 일시
- `Transfer-Encoding` — 본문 전송 인코딩 방식 (`chunked` 등)
- `Cache-Control` — 캐시 지시자

### 요청 헤더

- `Host` — 요청 대상 서버의 호스트명 (HTTP/1.1 필수)
- `User-Agent` — 클라이언트 애플리케이션 정보
- `Accept` — 클라이언트가 처리 가능한 미디어 타입
- `Accept-Encoding` — 처리 가능한 인코딩 (`gzip`, `deflate`)
- `If-Modified-Since` — 조건부 요청, 변경된 경우에만 응답
- `Authorization` — 인증 정보

### 응답 헤더

- `Server` — 서버 소프트웨어 정보
- `Set-Cookie` — 클라이언트에 쿠키 설정
- `WWW-Authenticate` — 401 응답 시 인증 방법 안내

### 엔터티 헤더 (본문에 대한 정보)

- `Content-Type` — 본문의 미디어 타입
- `Content-Length` — 본문 크기 (바이트)
- `Content-Encoding` — 본문 압축 방식
- `Last-Modified` — 리소스 마지막 수정 시각
- `ETag` — 리소스의 버전 식별자

### 확장 헤더

표준에 정의되지 않은 커스텀 헤더도 자유롭게 추가할 수 있다. 수신 측이 모르는 헤더는 무시하면 된다.

---

## 핵심 정리

> HTTP 메시지는 시작줄 + 헤더 + 본문의 단순한 텍스트 구조다. 요청줄에는 메서드·URL·버전이, 응답줄에는 버전·상태코드·사유구절이 온다. 메서드는 서버에게 원하는 동작을 알려주고, 상태 코드는 처리 결과를 알려준다. 헤더로 부가 정보를 전달한다.

# HTTP 완벽가이드 4장 — 커넥션 관리

## TCP 커넥션

HTTP 통신은 TCP 커넥션 위에서 이루어진다. 커넥션이 맺어지면 메시지가 손실·손상·순서 뒤바뀜 없이 전달된다고 믿을 수 있다. 이건 TCP가 보장해주는 것이지 HTTP가 하는 일이 아니다.

### 계층 구조

```
HTTP        ← 애플리케이션 계층
TLS/SSL     ← 보안 계층 (HTTPS일 때만)
TCP         ← 전송 계층
IP          ← 네트워크 계층
```

HTTP가 메시지를 전송하면 실제로는 TCP가 세그먼트 단위로 쪼개서 IP 패킷에 담아 보낸다. 이 과정은 HTTP 입장에서 투명하다.

### TCP 스트림은 세그먼트로, 세그먼트는 IP 패킷으로

TCP는 데이터를 **스트림**으로 다룬다. 바이트 단위로 흘려보내면 TCP가 알아서 세그먼트로 잘라서 IP 패킷에 담는다. 각 IP 패킷에는 출발지/목적지 IP, 출발지/목적지 포트, 제어 플래그 등이 들어간다.

### TCP 커넥션 식별

TCP 커넥션은 네 가지 값의 조합으로 유일하게 식별된다.

```
<발신지 IP> <발신지 포트> <수신지 IP> <수신지 포트>
```

이 네 값이 모두 같은 커넥션은 동시에 두 개 존재할 수 없다.

---

## TCP 성능

HTTP 트랜잭션의 대부분의 시간은 TCP 통신에 소비된다. 데이터 전송 자체보다 커넥션 수립과 관리에 드는 비용이 크다.

### HTTP 트랜잭션 지연의 원인들

1. **DNS 해석** — 호스트명 → IP 변환. 캐시가 없으면 수십~수백 ms 걸릴 수 있다.
2. **TCP 커넥션 핸드셰이크** — SYN → SYN+ACK → ACK. 작은 요청이라도 이 왕복이 필수다.
3. **요청 전송 + 서버 처리 시간**
4. **응답 전송**

작은 데이터를 주고받는 트랜잭션일수록 실제 전송 시간보다 핸드셰이크 비용이 상대적으로 크다.

### TCP의 성능 관련 특성들

**TCP 느린 시작 (Slow Start)** — 새 커넥션은 처음에 전송 속도를 낮게 시작해서 점진적으로 올린다. 네트워크 혼잡을 방지하기 위한 메커니즘이다. 이미 데이터를 주고받아 "튜닝된" 기존 커넥션이 새 커넥션보다 빠른 이유가 이것이다.

**네이글 알고리즘 (Nagle's Algorithm)** — 작은 데이터를 모아서 한 세그먼트에 담아 보낸다. 네트워크 효율은 좋아지지만, 작은 HTTP 메시지가 지연될 수 있다. 확인응답 지연과 만나면 최악의 조합이 된다. `TCP_NODELAY`로 끌 수 있다.

**확인응답 지연 (Delayed ACK)** — TCP는 ACK를 바로 보내지 않고 같은 방향 데이터에 편승시키려 한다. 편승할 데이터가 없으면 100~200ms 타이머가 만료될 때까지 기다린다. 이게 네이글 알고리즘과 맞물리면 불필요한 지연이 생긴다.

**TIME_WAIT 누적** — TCP 커넥션이 닫힌 뒤에도 같은 포트 조합의 패킷이 새 커넥션에 주입되는 것을 방지하기 위해 일정 시간(보통 2MSL, 약 2분) 동안 포트를 점유한다. 커넥션을 열고 닫기를 반복하면 TIME_WAIT 상태의 포트가 쌓인다.

## HTTP 커넥션 관리

### 순차적 커넥션 (Serial)

가장 단순한 방식. 하나의 트랜잭션이 끝나야 다음 트랜잭션을 시작한다. 느리다. 이미지가 10개인 페이지라면 10번 연속으로 커넥션을 맺고 끊는다.

### 병렬 커넥션 (Parallel)

여러 TCP 커넥션을 동시에 열어서 여러 트랜잭션을 병렬로 수행한다. 체감 속도가 빨라진다. 다만 커넥션을 너무 많이 열면 서버와 네트워크에 부하를 준다. 브라우저는 보통 도메인당 6~8개 정도로 제한한다.

병렬 커넥션이 항상 빠른 건 아니다. 대역폭이 충분하지 않으면 각 커넥션이 느려져서 이득이 없을 수 있다.

### 지속 커넥션 (Persistent Connection)

트랜잭션이 끝나도 커넥션을 닫지 않고 재사용한다. 핸드셰이크 비용을 아끼고, TCP 느린 시작이 이미 완료된 커넥션을 활용하니 성능이 좋아진다.

**HTTP/1.0+ Keep-Alive** — `Connection: Keep-Alive` 헤더로 지속 커넥션을 요청한다. 서버가 응답에도 같은 헤더를 넣으면 성립. `Keep-Alive: timeout=120, max=5`처럼 옵션도 줄 수 있다.

Keep-Alive의 문제점: 멍청한 프록시(blind relay)가 `Connection` 헤더를 이해하지 못하고 그대로 전달하면, 서버와 프록시 사이에 커넥션이 끊기지 않는 행(hang) 상태가 된다. 이게 **프록시와 홉별(hop-by-hop) 헤더 문제**다.

**HTTP/1.1 지속 커넥션** — 기본이 지속 커넥션이다. 끊고 싶으면 `Connection: close`를 명시해야 한다. Keep-Alive 헤더는 더 이상 사용하지 않는다.

### 파이프라인 커넥션 (Pipelining)

지속 커넥션 위에서 응답을 기다리지 않고 여러 요청을 연속으로 보내는 방식. 네트워크 왕복을 줄여준다.

제약 사항:
- 응답은 요청 순서대로 와야 한다 (FIFO)
- 비멱등 메서드(POST 등)는 파이프라이닝하면 안 된다. 실패 시 어디까지 처리됐는지 알 수 없다.
- 서버가 커넥션을 언제든 끊을 수 있으므로, 재시도 가능한 요청만 파이프라이닝해야 한다.
- 실제로는 HOL(Head-of-Line) 블로킹 문제 때문에 대부분의 브라우저가 파이프라이닝을 꺼 놓았다.

## 커넥션 끊기

TCP 커넥션은 입력 채널과 출력 채널이 독립적이다. 이 점을 이해해야 끊기 방식의 차이가 보인다.

### 전체 끊기 (Full Close)

`close()`를 호출하면 입력과 출력 채널을 모두 즉시 닫는다. 단순하지만, 상대방이 아직 데이터를 보내고 있었다면 그 데이터가 유실된다. 상대방은 "connection reset by peer" 에러를 받게 되고, 이미 버퍼에 있던 미처리 응답 데이터도 날아갈 수 있다.

### 절반 끊기 (Half-Close)

`shutdown()`을 사용해서 한쪽 채널만 닫는다.

- **출력만 끊기** — "나는 더 보낼 게 없다"는 신호를 상대에게 보낸다. 상대방은 이 신호를 받고도 자기 데이터를 마저 보낼 수 있다.
- **입력만 끊기** — 상대가 보내는 데이터를 더 이상 받지 않겠다는 뜻. 상대가 이후 데이터를 보내면 TCP 스택이 RST 패킷으로 응답하게 되어, 상대방의 나머지 파이프라인 데이터가 소실될 위험이 있다. 그래서 입력만 끊는 것은 위험하다.

### 우아한 커넥션 끊기 (Graceful Close)

안전하게 끊기 위한 관례적 절차다.

1. 자신의 출력 채널을 먼저 절반 끊기한다.
2. 상대방의 출력 채널이 닫히는 것을 기다리면서, 남은 데이터를 계속 읽는다.
3. 양쪽 모두 더 이상 보낼 데이터가 없다는 것이 확인되면 전체 끊기한다.

핵심은 출력을 먼저 끊고, 입력은 상대가 마무리할 때까지 열어두는 것이다.

### TCP RST (커넥션 리셋)

정상적인 FIN 기반 종료가 아니라, 한쪽이 RST 패킷을 보내서 커넥션을 강제로 끊는 것이다. 입력 채널을 먼저 닫은 상태에서 상대가 데이터를 보내면 발생할 수 있다. RST를 받은 쪽은 읽기 버퍼에 쌓여 있던 데이터까지 전부 폐기하게 되므로, 이미 도착해 있었지만 아직 읽지 않은 응답이 사라지는 문제가 생긴다.

### Content-Length와 커넥션 끊기의 관계

서버가 `Content-Length`를 정확히 명시하면 클라이언트는 본문의 끝을 정확히 알 수 있다. 하지만 `Content-Length`가 없거나 `Transfer-Encoding: chunked`도 아닌 경우, 클라이언트는 서버가 커넥션을 끊는 시점을 본문의 끝으로 간주한다. 이 때문에 커넥션이 의도치 않게 끊기면 불완전한 데이터를 정상으로 오인할 수 있다.

### 멱등성과 재시도

커넥션이 예기치 않게 끊겼을 때, 클라이언트는 해당 요청을 재시도할 수 있는지 판단해야 한다. GET, HEAD, PUT, DELETE 같은 멱등 메서드는 여러 번 실행해도 결과가 같으므로 재시도해도 안전하다. 반면 POST 같은 비멱등 메서드는 재시도하면 중복 처리가 발생할 수 있어서 함부로 재시도하면 안 된다. 파이프라이닝에서 비멱등 메서드를 쓰지 말라는 제약도 여기서 나온다.

---

## 핵심 정리

> HTTP는 TCP 위에서 동작하고, 트랜잭션 지연의 상당 부분은 TCP 핸드셰이크와 느린 시작 때문이다. 지속 커넥션으로 커넥션 재사용, 병렬 커넥션으로 동시 요청, 파이프라이닝으로 왕복 절감을 할 수 있다. HTTP/1.1은 기본이 지속 커넥션이며, 끊을 때는 `Connection: close`를 명시한다.

# HTTP 완벽가이드 6장 — 프락시

## 프락시란

클라이언트와 서버 사이에 위치해서 HTTP 트래픽을 중계하는 중간 서버다. 클라이언트의 요청을 받아서 서버에 전달하고, 서버의 응답을 다시 클라이언트에 돌려준다. 이 과정에서 요청이나 응답을 수정할 수도 있다.

프락시는 단순히 메시지를 전달만 하는 게 아니라, 필터링·캐싱·로깅·인증·로드밸런싱 등 다양한 부가 기능을 수행한다. 웹 생태계의 핵심 구성요소 중 하나다.

---

## 포워드 프락시 vs 리버스 프락시

이 구분이 프락시 이해의 출발점이다.

### 포워드 프락시 (Forward Proxy)

**클라이언트 측에 위치**한다. 클라이언트가 인터넷에 접근할 때 거쳐가는 관문 역할이다.

```
[클라이언트] → [포워드 프락시] → [인터넷/서버]
```

주요 용도:
- 클라이언트의 IP를 숨겨 익명성을 제공한다
- 조직 내 콘텐츠 필터링(특정 사이트 차단)
- 캐싱을 통한 대역폭 절약
- 사내 네트워크에서 외부 접근 통제

기업, 학교 같은 조직에서 많이 사용한다. 최근에는 원격 근무 확대와 함께 클라우드 기반 포워드 프락시(Zscaler 같은 SaaS 형태)가 늘고 있다.

### 리버스 프락시 (Reverse Proxy)

**서버 측에 위치**한다. 클라이언트는 리버스 프락시와 통신하지만, 실제 요청은 뒤에 있는 원 서버(origin server)로 전달된다. 클라이언트는 리버스 프락시의 존재를 모를 수도 있다.

```
[클라이언트] → [리버스 프락시] → [원 서버(들)]
```

주요 용도:
- 로드밸런싱 — 여러 서버에 트래픽을 분산
- SSL/TLS 종료 — 암호화·복호화를 프락시가 처리해서 백엔드 서버 부하를 줄임
- 캐싱 — 정적 콘텐츠를 프락시 레벨에서 제공
- 보안 — 백엔드 서버의 IP를 숨기고, DDoS 방어·WAF 기능 수행
- 압축 — 응답 데이터를 gzip/brotli로 압축

Nginx, HAProxy가 전통적인 리버스 프락시 도구이고, 현재는 Cloudflare, AWS CloudFront 같은 CDN이 리버스 프락시 역할을 겸한다.

---

## 프락시가 트래픽을 처리하는 방식

### 프락시 vs 게이트웨이

- **프락시** — 같은 프로토콜의 양쪽을 중계한다 (HTTP ↔ HTTP)
- **게이트웨이** — 서로 다른 프로토콜을 변환한다 (HTTP ↔ DB 프로토콜 등)

실무에서는 이 경계가 모호하다. 하나의 장비가 프락시이면서 게이트웨이 역할을 동시에 하는 경우가 많다.

### 프락시가 트래픽을 받는 방법

클라이언트 트래픽이 프락시를 거치게 하려면 네 가지 방법이 있다.

**1. 클라이언트 설정** — 브라우저나 OS에서 프락시 주소를 직접 설정한다. 클라이언트가 프락시의 존재를 인지하고, 요청을 프락시에게 보낸다.

**2. 네트워크 인터셉트 (투명 프락시)** — 네트워크 장비(라우터, 스위치)가 클라이언트 모르게 트래픽을 가로채서 프락시로 보낸다. 클라이언트는 프락시의 존재를 모른다.

**3. 대리(서로게이트) 프락시** — 리버스 프락시의 다른 이름이다. DNS가 원 서버 대신 프락시의 IP를 반환하도록 설정한다. 클라이언트는 원 서버에 접속한다고 생각하지만 실제로는 프락시와 통신한다.

**4. PAC (Proxy Auto-Configuration)** — JavaScript 파일로 프락시 설정을 동적으로 결정한다. URL 패턴에 따라 다른 프락시를 사용하거나, 프락시를 우회할 수 있다. WPAD(Web Proxy Auto-Discovery) 프로토콜로 PAC 파일의 위치를 자동 탐색하기도 한다.

## 프락시가 하는 일들

### 콘텐츠 필터링

조직 내에서 부적절한 사이트 차단, 악성 콘텐츠 차단 등을 수행한다. 성인 사이트 차단기가 대표적인 예다.

### 접근 제어

프락시가 인증·인가를 담당한다. 클라이언트가 리소스에 접근하려면 프락시에서 먼저 자격 증명을 통과해야 한다. 407 Proxy Authentication Required 응답과 `Proxy-Authorization` 헤더가 이 과정에서 사용된다.

### 캐싱

자주 요청되는 리소스의 사본을 프락시에 저장해서 원 서버까지 갈 필요 없이 바로 응답한다. 대역폭 절약과 응답 속도 향상에 효과적이다. 캐시에 대해서는 7장에서 상세히 다룬다.

### 로드밸런싱

들어오는 요청을 여러 서버에 분산시킨다. 라운드 로빈, 최소 연결, 가중치 기반 등 다양한 알고리즘을 사용한다.

### 로깅과 감사

프락시를 거치는 모든 트래픽을 기록해서 사용 패턴을 분석하거나 감사 추적(audit trail)에 활용한다.

### 콘텐츠 변환

응답 본문을 실시간으로 변환한다. 이미지 압축, 텍스트 인코딩 변환, HTML 축소(minification) 같은 작업을 프락시 레벨에서 수행할 수 있다.

### 익명화

클라이언트의 IP 주소, `User-Agent`, `Referer`, `Cookie` 같은 식별 정보를 제거하거나 변조해서 사용자의 프라이버시를 보호한다.

## 프락시와 HTTP 헤더

### Via 헤더

메시지가 어떤 프락시를 거쳐갔는지 추적하는 헤더다. 프락시를 거칠 때마다 자신의 정보를 Via 헤더에 추가한다.

```
Via: 1.1 proxy-a.example.com, 1.0 proxy-b.example.com
     ─── ──────────────────
    버전    프락시 호스트
```

요청과 응답 양쪽 모두에 사용된다. 무한 루프 탐지와 프로토콜 능력 파악에 도움을 준다.

### X-Forwarded-For

원래 클라이언트의 IP 주소를 전달하는 헤더다. 프락시를 거치면 서버가 보는 IP가 프락시의 IP로 바뀌는데, 원래 클라이언트 IP를 알아야 할 때 이 헤더를 참조한다.

```
X-Forwarded-For: 203.0.113.50, 70.41.3.18
                 ─────────────  ──────────
                 원래 클라이언트  첫 번째 프락시
```

> **2026 기준 참고**: `X-Forwarded-For`는 비표준 헤더였으나, RFC 7239에서 `Forwarded` 헤더가 표준으로 정의되었다. `Forwarded: for=203.0.113.50;proto=https;by=70.41.3.18` 형태를 사용한다. 다만 실무에서는 하위 호환성 때문에 `X-Forwarded-For`가 여전히 널리 쓰인다. Spring Boot에서도 `server.forward-headers-strategy=native` 설정 시 두 헤더 모두 인식한다.

### X-Forwarded-Host / X-Forwarded-Proto

- `X-Forwarded-Host` — 원래 클라이언트가 요청한 호스트명
- `X-Forwarded-Proto` — 원래 요청의 프로토콜 (http / https)

리버스 프락시 뒤에서 SSL 종료를 하면, 백엔드 서버는 평문 HTTP 요청만 받게 된다. 이때 원래 요청이 HTTPS였다는 것을 `X-Forwarded-Proto`로 알 수 있다. 리다이렉트 URL 생성이나 쿠키의 Secure 플래그 판단에 필수적이다.

### 홉별(Hop-by-Hop) 헤더 vs 종단간(End-to-End) 헤더

- **홉별 헤더** — 하나의 프락시 구간에서만 유효. 다음 홉으로 전달하면 안 된다. `Connection`, `Keep-Alive`, `Proxy-Authorization`, `TE`, `Transfer-Encoding`, `Upgrade` 등.
- **종단간 헤더** — 최종 수신자까지 전달되어야 한다. `Cache-Control`, `Content-Type` 등 대부분의 헤더.

`Connection` 헤더에 명시된 헤더명은 홉별 헤더로 취급되어 다음 프락시로 전달되지 않아야 한다. 4장에서 다뤘던 Keep-Alive 프록시 문제가 바로 이 규칙을 위반한 멍청한 프록시(dumb proxy) 때문에 발생한다.

## 프락시와 요청 URI

### 클라이언트가 프락시에 보내는 요청의 차이

프락시에게 보내는 요청은 일반 서버 요청과 형태가 다르다.

```
# 서버에 직접 요청 — 부분 URI
GET /index.html HTTP/1.1
Host: www.example.com

# 프락시에 요청 — 완전한(absolute) URI
GET http://www.example.com/index.html HTTP/1.1
```

프락시는 완전한 URI를 받아야 어디로 전달할지 알 수 있다. 반면 서버에 직접 보낼 때는 Host 헤더로 목적지를 알 수 있으니 경로만 보내면 된다.

### 투명 프락시에서의 문제

투명 프락시는 클라이언트가 프락시의 존재를 모르기 때문에 부분 URI를 보낸다. 프락시는 Host 헤더나 인터셉트한 네트워크 정보에서 목적지를 알아내야 한다. Host 헤더가 없는 HTTP/1.0 요청이라면 DNS 역방향 조회 같은 방법에 의존해야 하는데, 이건 불완전하다.

### URI 해석 (URI Resolution)

프락시는 URI를 파싱해서 스킴, 호스트, 포트, 경로를 추출하고, 목적지 서버를 결정한다. 이 과정에서 호스트명이 IP로 해석 가능한지, 직접 연결할지 다른 부모 프락시로 넘길지 등을 판단한다.

---

## 프락시 체이닝과 계층

프락시는 여러 개가 연쇄적으로 연결될 수 있다. 이를 프락시 체인이라고 한다.

```
[클라이언트] → [프락시 A] → [프락시 B] → [프락시 C] → [서버]
```

### 계층 구조

조직 네트워크에서는 프락시를 계층적으로 배치한다. 하위 프락시(자식)가 처리할 수 없는 요청을 상위 프락시(부모)로 올려보내는 구조다.

### 동적 부모 선택

요청의 내용에 따라 다른 부모 프락시로 라우팅할 수 있다. 예를 들어 특정 도메인은 캐시 프락시로, 유료 콘텐츠는 인증 프락시로 보내는 식이다. 부하가 분산되고 유연한 라우팅이 가능하다.

## 2026 기준 — 현대 프락시 생태계

책의 원서가 쓰인 시점과 비교해서 프락시의 역할과 형태가 크게 진화했다.

### CDN이 사실상의 리버스 프락시

Cloudflare, AWS CloudFront, Akamai 같은 CDN이 전 세계 엣지 서버에서 리버스 프락시 역할을 수행한다. 캐싱, SSL 종료, DDoS 방어, WAF, 봇 탐지를 CDN 레이어에서 통합 처리한다. 중소 규모 서비스라도 CDN을 앞에 두는 것이 사실상 표준이 되었다. 전 세계 웹사이트의 상당수가 Cloudflare나 Nginx 기반 리버스 프락시를 사용한다.

### API 게이트웨이

마이크로서비스 아키텍처에서 클라이언트와 내부 서비스 사이의 단일 진입점 역할을 하는 특수한 리버스 프락시다. 인증, 속도 제한(rate limiting), 요청 라우팅, 프로토콜 변환, API 버전 관리 등을 담당한다.

대표적인 도구:
- **Spring Cloud Gateway** — Spring 생태계의 API 게이트웨이. 필터 체인 기반으로 요청·응답을 변환한다.
- **Kong** — Nginx 기반의 오픈소스 API 게이트웨이. 플러그인 시스템이 강점이다.
- **AWS API Gateway** — 관리형 서비스. 서버리스(Lambda) 아키텍처와 궁합이 좋다.

### 서비스 메시와 사이드카 프락시

마이크로서비스 간 통신을 관리하는 인프라 레이어다. 각 서비스 인스턴스 옆에 사이드카 프락시를 배치해서, 서비스 코드 변경 없이 네트워킹 관심사(라우팅, 보안, 관찰성)를 처리한다.

```
[서비스 A] ↔ [Envoy 사이드카] ↔ ··· ↔ [Envoy 사이드카] ↔ [서비스 B]
                        ↕
                  [컨트롤 플레인 (Istio)]
```

**Envoy Proxy** — Lyft에서 개발하고 CNCF에서 졸업한 고성능 L3/L4/L7 프락시다. Nginx, HAProxy 같은 전통 프락시와 달리, 동적 설정 변경이 가능하고 HTTP/2·gRPC를 네이티브 지원한다. 서비스 메시의 데이터 플레인 표준으로 자리 잡았다.

**Istio** — Envoy 사이드카를 관리하는 컨트롤 플레인이다. 트래픽 관리(카나리 배포, A/B 테스트), mTLS(상호 TLS), 분산 트레이싱, 서킷 브레이커 같은 기능을 서비스 코드와 분리해서 제공한다.

전통적인 프락시(6장의 프락시)가 네트워크 경계에서 동작했다면, 사이드카 프락시는 각 서비스 인스턴스 레벨에서 동작한다는 점이 가장 큰 차이다.

### Zero Trust 프락시

기존의 네트워크 경계 기반 보안("성벽 안은 안전하다")이 무너지면서, 모든 요청을 인증·인가하는 Zero Trust 모델이 부상했다. 프락시가 이 모델의 핵심 실행 지점이 된다.

- 모든 요청에 대해 사용자 신원과 디바이스 상태를 검증한다
- 네트워크 위치가 아니라 요청 단위로 접근을 허용한다
- Cloudflare Access, Google BeyondCorp, Zscaler 같은 서비스가 대표적이다

### HTTP/2, HTTP/3과 프락시

6장이 쓰인 시점에는 HTTP/1.1이 주력이었지만, 현재는 HTTP/2가 보편적이고 HTTP/3(QUIC 기반)도 빠르게 확산 중이다.

- **HTTP/2** — 멀티플렉싱으로 HOL 블로킹을 해소하고, 헤더 압축(HPACK)으로 오버헤드를 줄인다. 프락시가 HTTP/2로 클라이언트와 통신하면서 백엔드에는 HTTP/1.1이나 HTTP/2로 중계하는 것이 일반적이다.
- **HTTP/3** — UDP 기반의 QUIC 프로토콜을 사용한다. 커넥션 수립이 빠르고, TCP 레벨의 HOL 블로킹까지 해소한다. CDN(Cloudflare, Fastly)이 HTTP/3 지원을 선도하고 있고, 프락시가 이 프로토콜 전환의 중간 역할을 한다.

---

## 핵심 정리

> 프락시는 클라이언트와 서버 사이에서 HTTP 트래픽을 중계하는 중간 서버다. 포워드 프락시는 클라이언트 측, 리버스 프락시는 서버 측에 위치한다. Via 헤더로 프락시 경유를 추적하고, 홉별 헤더는 다음 프락시로 전달하면 안 된다. 현대 웹에서는 CDN이 리버스 프락시를 겸하고, API 게이트웨이와 서비스 메시의 사이드카 프락시가 마이크로서비스 통신을 담당하며, Zero Trust 모델에서 프락시는 모든 요청의 인증·인가 실행 지점이 되었다.

# HTTP 완벽가이드 7장 — 캐시

## 캐시가 필요한 이유

캐시는 자주 사용되는 리소스의 사본을 클라이언트와 가까운 위치에 저장해두고 재사용하는 메커니즘이다. HTTP는 본질적으로 요청-응답 기반이고, 네트워크 왕복(round trip)이 가장 큰 비용이다. 캐시는 이 왕복 자체를 없애거나 줄이는 방식으로 동작한다.

캐시가 해결하는 문제는 다섯 가지다.

### 1. 불필요한 데이터 전송 제거

같은 응답을 매번 원 서버에서 받지 않는다. 동일한 사본을 여러 클라이언트가 재사용하면 그만큼 중복 전송이 사라진다.

### 2. 네트워크 병목 완화

대역폭이 좁은 네트워크에서도 페이지가 빠르게 뜨는 이유는 캐시 때문이다. 데이터를 다시 전송할 필요가 없으니 회선 용량을 다른 곳에 쓸 수 있다.

### 3. 원 서버 요구 감소

캐시가 트래픽을 흡수해서 원 서버까지 가는 요청 자체가 줄어든다. 같은 비용으로 더 많은 사용자를 처리할 수 있다.

### 4. 거리로 인한 지연 감소

지구 반대편의 서버에 요청을 보내면 빛의 속도라는 물리 법칙 때문에 최소 100~200ms는 걸린다. 캐시가 사용자 근처에 있으면 이 지연이 거의 사라진다. 이것이 CDN이 효과적인 근본 이유다.

### 5. 갑작스러운 트래픽 폭주 완화

이벤트로 트래픽이 폭증할 때 캐시가 방파제 역할을 한다. 원 서버 한 대가 처리 못 할 트래픽을 캐시 계층이 흡수한다.

## 캐시 적중과 부적중

### 캐시 적중 (Cache Hit)

요청에 해당하는 사본이 캐시에 있고, 그대로 사용 가능한 상태다. 원 서버에 가지 않고 캐시가 직접 응답한다. 가장 빠르고 효율적인 경로다.

### 캐시 부적중 (Cache Miss)

캐시에 사본이 없는 경우. 캐시는 원 서버에 요청을 보내서 응답을 받아오고, 클라이언트에 전달하면서 동시에 자신의 저장소에도 보관한다. 다음 요청부터는 적중이 가능하다.

### 재검사 적중 (Revalidate Hit / Slow Hit)

캐시에 사본은 있지만 신선한지 확실하지 않은 경우다. 캐시는 원 서버에 "내 사본이 아직 유효한가?"라고 조건부 요청을 보낸다.

- 유효하면 → 서버가 `304 Not Modified` 응답 (본문 없음) → 캐시는 자기 사본을 그대로 사용
- 유효하지 않으면 → 서버가 `200 OK` 응답 (새 본문 포함) → 캐시는 사본을 갱신

본문이 전송되지 않을 때가 많아서 대역폭은 여전히 절약된다. 다만 왕복은 발생하므로 완전 적중보다는 느리다.

### 적중률 계산

| 지표 | 정의 |
|------|------|
| 문서 적중률 (Document Hit Rate) | 캐시가 처리한 요청 수 / 전체 요청 수 |
| 바이트 적중률 (Byte Hit Rate) | 캐시가 전송한 바이트 / 전체 바이트 |

두 지표가 다른 이유는 캐시 적중되는 리소스의 크기가 제각각이기 때문이다. 작은 HTML은 자주 적중되지만 바이트는 작고, 큰 비디오는 가끔 적중돼도 바이트는 크다. 대역폭 절약을 측정할 때는 바이트 적중률이, 응답 속도를 측정할 때는 문서 적중률이 더 의미 있다.

## 캐시 처리 단계

캐시가 요청을 받았을 때 거치는 표준 흐름이다. 책에서 7단계로 정리된다.

### 1단계 — 요청 받기

클라이언트로부터 HTTP 요청 메시지를 수신한다. 네트워크 인터페이스에서 메시지를 읽어들이는 단계다.

### 2단계 — 파싱

요청 메시지를 파싱해서 메서드, URL, 헤더값들을 자료구조에 담는다.

### 3단계 — 검색

URL을 캐시 키로 사용해서 저장소에서 사본을 찾는다. 사본이 없으면 부적중으로 분기. `Vary` 헤더가 관련된 경우 헤더값까지 키로 사용한다.

### 4단계 — 신선도 검사

사본이 있으면 충분히 신선한지 확인한다. 신선하면 6단계로, 아니면 5단계로.

### 5단계 — 서버 재검사

만료된 사본을 가지고 원 서버에 조건부 요청을 보낸다. `304`면 사본을 그대로 쓰면서 헤더만 갱신, `200`이면 새 사본으로 교체한다.

### 6단계 — 응답 생성

캐시 사본을 기반으로 응답 메시지를 만든다. `Age`, `Via` 같은 헤더를 추가하거나 수정한다.

### 7단계 — 전송

클라이언트에 응답을 보낸다.

### 8단계 — 로깅

트랜잭션 기록을 남긴다. 적중/부적중 통계, 처리 시간, 트래픽 등을 추적한다.

## 신선도와 만료 (Freshness & Expiration)

캐시 사본이 언제까지 유효한지 결정하는 기준이다. 두 가지 메커니즘이 있고, 둘 다 응답 헤더로 지정된다.

### Expires 헤더 (HTTP/1.0)

응답이 만료되는 절대 시각을 GMT 시간으로 지정한다.

```
Expires: Thu, 31 Dec 2026 23:59:59 GMT
```

**문제점**: 서버와 캐시 컴퓨터의 시계가 어긋나면 정확하지 않다. 시계가 동기화돼 있어야 작동하는데, 분산된 환경에서 이걸 보장하기 어렵다. 또 응답이 캐시까지 도달하는 시간을 무시한다.

### Cache-Control: max-age (HTTP/1.1)

응답이 생성된 시점부터 몇 초 동안 신선한지를 **상대 시간**으로 지정한다.

```
Cache-Control: max-age=3600
```

응답이 생성되고 3600초(1시간) 동안 신선하다는 의미다. 시계 동기화가 필요 없어서 분산 환경에서도 안정적이다. **현재 표준은 이 방식**이고, `Expires`보다 우선한다.

### 신선도 수명 vs 나이

캐시는 두 값을 계산하고 비교한다.

- **신선도 수명 (Freshness Lifetime)** — 응답이 유효한 총 시간. `max-age`나 `Expires - Date`로 계산된다.
- **나이 (Age)** — 응답이 원 서버에서 생성된 후 흐른 시간.

```
나이 < 신선도 수명  →  신선함
나이 ≥ 신선도 수명  →  만료됨 (재검사 필요)
```

### 휴리스틱 만료 (Heuristic Expiration)

`Cache-Control: max-age`도 `Expires`도 없으면 캐시는 어떻게 신선도를 판단할까? 휴리스틱(경험적 규칙)을 사용한다.

`Last-Modified`가 있으면 그 시각으로부터 흐른 시간의 **10%를 신선도 수명**으로 잡는 것이 일반적이다. 예를 들어 5일 전에 수정된 파일은 12시간 정도 신선하다고 본다. 단 이 휴리스틱은 캐시 구현마다 다르고, 24시간을 넘기지 않도록 권장된다. 휴리스틱 기반 신선도가 24시간을 넘으면 캐시는 `Warning: 113` 헤더를 응답에 추가해야 한다.

명확한 신선도 정책이 필요하면 항상 `Cache-Control`이나 `Expires`를 명시해야 한다.

## Age 헤더와 나이 계산

응답이 캐시에서 얼마나 오래됐는지를 초 단위로 표현하는 헤더다. 응답이 여러 캐시를 거치면서 각 캐시가 머문 시간이 누적된다.

```
Age: 1842
```

### Date vs Age

- `Date` — 원 서버가 응답을 생성한 시각 (절대 시간)
- `Age` — 그 시점부터 지금까지 흐른 시간 (상대 시간)

### 나이 계산이 복잡한 이유

이론적으로 `나이 = 현재시각 - Date`이면 되지만, 실제로는 다음을 보정해야 한다.

1. 응답이 원 서버에서 출발해서 캐시에 도달하기까지의 전송 지연
2. 캐시 간에 응답이 전달될 때마다 발생하는 지연
3. 캐시 내부에 머문 시간

책에서는 `apparent_age`, `corrected_age`, `resident_time` 같은 개념으로 이를 보정한다. 핵심은 **나이를 과소평가하지 말고 보수적으로 추정**해서, 만료된 응답을 신선하다고 잘못 판단하지 않는 것이다.

## 서버 재검사 (Server Revalidation)

만료된 사본이 있을 때, 본문을 다시 받지 않고 헤더만으로 유효성을 확인하는 과정이다. 두 가지 검사기(validator)를 사용한다.

### Last-Modified와 If-Modified-Since

서버가 응답할 때 리소스의 최종 수정 시각을 `Last-Modified` 헤더로 알려준다.

```
응답: Last-Modified: Tue, 28 May 2026 12:00:00 GMT
```

캐시는 재검사 시 이 값을 `If-Modified-Since`에 담아 다시 보낸다.

```
요청: If-Modified-Since: Tue, 28 May 2026 12:00:00 GMT

응답: 304 Not Modified           (변경 없음 → 캐시 사용)
     또는
     200 OK + 새 본문              (변경됨 → 새 사본 받아 갱신)
```

**한계점**:
- 초 단위 정밀도라서 1초 안에 여러 번 수정되면 변경을 감지 못한다
- 콘텐츠는 그대로지만 메타데이터만 바뀐 경우도 변경으로 잡힌다 (불필요한 재전송)
- 정확한 시각 기록이 어려운 동적 콘텐츠에는 부적합하다

### ETag와 If-None-Match

ETag는 리소스의 버전을 식별하는 **임의의 문자열**이다. 시간이 아니므로 위의 한계가 없다. 서버가 생성 규칙을 자유롭게 정할 수 있다(내용 해시, 버전 번호, 수정 카운터 등).

```
응답: ETag: "v2.3-abc123"

재검사 요청: If-None-Match: "v2.3-abc123"

응답: 304 Not Modified  (일치 → 캐시 사용)
     또는
     200 OK + 새 본문    (불일치 → 새 사본)
```

### 강한 검사기 vs 약한 검사기

- **강한 검사기 (Strong Validator)** — 바이트 단위로 정확히 일치할 때만 같은 것으로 판단한다. 기본 ETag 형식.
- **약한 검사기 (Weak Validator)** — 의미상 동등하면 같은 것으로 판단한다. ETag 앞에 `W/` 접두어를 붙인다.

```
강한: ETag: "v2.3"
약한: ETag: W/"v2.3"
```

| 검사기 | 용도 | 예시 |
|--------|------|------|
| 강한 | 부분 요청, 캐시 무결성이 중요한 상황 | 파일 다운로드 이어받기 |
| 약한 | 콘텐츠 의미가 같으면 OK인 상황 | 배너 광고가 매번 다른 일반 HTML |

`Last-Modified`는 본질적으로 약한 검사기에 가깝다(초 단위 정밀도 한계 때문).

### 둘 다 있으면

서버가 응답에 `Last-Modified`와 `ETag`를 모두 보냈다면, 캐시는 재검사 시 둘 다 보내는 것이 권장된다. HTTP/1.1 서버는 ETag를 우선 사용해서 판단해야 한다.

```
If-Modified-Since: Tue, 28 May 2026 12:00:00 GMT
If-None-Match: "v2.3-abc123"
```

## Cache-Control 지시자 상세

### 응답 지시자 (서버 → 캐시)

**`max-age=N`** — N초 동안 신선함. 가장 자주 사용되는 지시자다.

**`s-maxage=N`** — 공유 캐시(프락시, CDN)에만 적용되는 max-age. 개인 캐시는 무시한다. 예: `Cache-Control: max-age=60, s-maxage=300` — 브라우저는 1분, CDN은 5분 캐싱.

**`public`** — 응답을 모든 캐시(공유 캐시 포함)에 저장 가능. 기본적으로 인증된 응답은 공유 캐시에 저장되지 않는데, `public`이 이걸 허용한다.

**`private`** — 응답을 개인 캐시(브라우저)에만 저장 가능. 공유 캐시(프락시, CDN)는 저장 금지. 사용자별 콘텐츠에 사용.

**`no-cache`** — 저장은 허용하되, **사용 전에 반드시 재검사**해야 한다. 이름이 헷갈리는데, "캐시하지 마라"가 아니라 "검증 없이 쓰지 마라"는 뜻이다.

**`no-store`** — 응답을 어디에도 저장하지 마라. 디스크, 메모리 모두 금지. 민감한 데이터에 사용.

**`must-revalidate`** — 만료된 사본은 절대 재검사 없이 쓰면 안 된다. 일부 캐시는 네트워크 장애 시 만료된 사본을 임시로 사용하는데, 이 지시자는 그것까지 금지한다.

**`proxy-revalidate`** — must-revalidate인데 공유 캐시에만 적용된다.

**`immutable`** — 응답이 신선한 동안은 절대 변하지 않는다는 약속. 브라우저가 사용자의 새로고침에서도 재검사를 생략한다. 파일명에 해시가 있는 정적 자산에 적합하다.

### 요청 지시자 (클라이언트 → 캐시)

**`max-age=N`** — N초보다 오래된 사본은 거부.

**`min-fresh=N`** — 앞으로 최소 N초 동안은 신선한 사본만 받아들임. 데이터를 받고 잠시 후에도 유효하길 원할 때.

**`max-stale=N`** — 만료된 지 N초 이내라면 받아들임. 신선도를 약간 양보해서 응답 속도를 우선시할 때.

**`no-cache`** — 캐시 사본을 그대로 쓰지 말고 재검사해라. 브라우저의 강제 새로고침에서 사용된다.

**`no-store`** — 이 요청과 응답을 저장하지 마라.

**`only-if-cached`** — 캐시에 있을 때만 응답해라. 없으면 `504 Gateway Timeout`. 오프라인 모드에서 캐시된 콘텐츠만 사용할 때.

### no-cache vs no-store

가장 자주 헷갈리는 두 지시자다.

| 지시자 | 저장 | 사용 |
|--------|------|------|
| `no-cache` | O | 매번 재검사 후 사용 |
| `no-store` | X | 저장 자체 금지 |

민감한 정보(인증 토큰, 결제 정보, 개인정보)는 반드시 `no-store`를 써야 한다. `no-cache`만 쓰면 디스크에 평문으로 저장되어 다른 사용자가 볼 수 있다.

### 확장 지시자 (책 출간 이후 추가됨)

- **`stale-while-revalidate=N`** — 만료된 사본을 N초 동안은 그대로 응답하면서, 백그라운드에서 재검사한다. 사용자는 빠른 응답을 받고, 다음 요청부터는 최신 데이터를 받게 된다.
- **`stale-if-error=N`** — 원 서버에 에러가 발생하면 N초 동안은 만료된 사본으로라도 응답한다. 가용성을 높인다.

이 둘은 RFC 5861에 정의됐고, CDN에서 활발히 사용된다.

## Vary 헤더

같은 URL이라도 요청 헤더에 따라 응답이 달라질 수 있다.

- `Accept-Encoding`에 따라 gzip 압축본 / 원본
- `Accept-Language`에 따라 한국어 / 영어 페이지
- `User-Agent`에 따라 모바일 / 데스크톱 페이지

`Vary` 헤더는 캐시에게 "이 헤더들이 다르면 다른 응답으로 취급하라"고 알려준다.

```
Vary: Accept-Encoding, Accept-Language
```

캐시는 URL뿐 아니라 `Vary`에 명시된 헤더값까지 **캐시 키에 포함**시킨다. 이를 통해 한국어 사용자에게 영어 페이지를 잘못 보여주는 사고를 막는다.

### Vary 헤더의 함정

- `Vary: *` — 어떤 요청 헤더라도 응답에 영향을 줄 수 있다는 뜻. 사실상 캐싱을 무력화한다.
- `Vary: User-Agent` — User-Agent는 종류가 너무 많아서 캐시 효율이 급격히 떨어진다. 디바이스 분기는 서버 사이드에서 처리하고 단순화된 헤더(`Vary: X-Device-Type`)를 사용하는 게 낫다.

`Vary`를 잘못 쓰면 캐시 히트율이 떨어지므로, 정말 필요한 헤더만 명시해야 한다.

---

## 캐시 종류와 토폴로지

### 개인 캐시 (Private Cache)

브라우저 캐시처럼 한 사용자에게만 속하는 캐시다. 크기는 작지만 비용이 거의 들지 않고, 사용자별 콘텐츠(로그인 후 페이지 등)도 저장할 수 있다.

### 공유 캐시 (Public / Shared Cache)

여러 사용자가 함께 사용하는 캐시다. 프락시 캐시, 게이트웨이 캐시, CDN 등이 여기 속한다. 같은 사본을 여러 사용자가 재사용하므로 효율이 매우 높다.

대신 사용자별 데이터를 저장하면 안 된다. `Cache-Control: private`로 공유 캐시 저장을 막을 수 있다.

### 프락시 캐시의 계층 구조

캐시는 계층적으로 배치된다.

```
[브라우저 캐시] → [회사 프락시 캐시] → [지역 프락시 캐시] → [CDN 엣지] → [원 서버]
```

가까운 캐시에서 부적중이 나면 다음 계층 캐시로 이동한다. 각 계층이 트래픽을 흡수해서 위로 올라갈수록 부하가 줄어든다.

### 캐시 메쉬 (Cache Mesh)

같은 계층의 캐시들이 서로 통신하면서 사본을 공유하는 구조다. 한 캐시에 없는 사본을 형제 캐시에게 물어볼 수 있다. ICP(Internet Cache Protocol), HTCP 같은 프로토콜이 있다.

---

## 캐시 무효화 (Invalidation)

원 서버에서 리소스가 변경되었을 때, 캐시된 사본을 어떻게 무효화할지가 중요한 문제다.

### HTTP 메서드와 자동 무효화

상태를 변경하는 메서드(`POST`, `PUT`, `DELETE`, `PATCH`)가 어떤 URL에 도달하면, 캐시는 그 URL과 관련된 사본을 무효화해야 한다.

```
POST /articles/123        →  GET /articles/123 캐시 무효화
DELETE /users/42          →  GET /users/42 캐시 무효화
```

또 응답의 `Location`, `Content-Location` 헤더에 명시된 URL도 같이 무효화 대상이다.

### 무효화의 본질적 한계

이 무효화는 **그 캐시를 거쳐간 요청에 한해서만** 일어난다. 다른 경로의 캐시는 변경을 알 수 없다. 예를 들어 사용자 A의 POST가 회사 프락시를 거쳤다면 그 프락시는 사본을 무효화하지만, 다른 회사의 프락시는 변경을 모른다.

분산 환경에서 완벽한 캐시 무효화는 본질적으로 어려운 문제다. 그래서 실무에서는 다음 방법들을 조합한다.

- **짧은 TTL** — 어차피 빨리 만료되니까 무효화 부담이 적다
- **명시적 퍼지(purge)** — CDN의 관리 API로 특정 URL의 캐시를 강제 삭제
- **버전 URL** — 콘텐츠 변경 시 URL 자체를 바꿔서 캐시 키를 달리한다

---

## 캐시 무력화 전략 (Cache Busting)

브라우저가 오래된 정적 자산을 사용하는 문제를 막기 위한 패턴들이다.

### 1. URL에 해시 포함 (Hash-based)

파일 내용의 해시를 URL에 넣는다. 내용이 바뀌면 URL이 바뀌므로 캐시 키가 자동으로 갱신된다.

```
app.abc123.js   →   app.def456.js  (내용 변경 시)
```

Webpack, Vite 같은 현대 빌드 도구가 기본 지원한다. 이 패턴을 쓰면 `Cache-Control: public, max-age=31536000, immutable`로 1년 캐싱해도 안전하다.

### 2. 쿼리 파라미터 (Query-based)

```
app.js?v=1.2.3
```

쉽지만 일부 프락시 캐시가 쿼리가 있는 URL을 캐시하지 않는 경우가 있어서, 해시 방식이 더 권장된다.

### 3. 디렉토리 버전 (Directory-based)

```
/static/v123/app.js
```

배포마다 디렉토리를 바꾸는 방식. 단순하지만 모든 자산이 한꺼번에 무효화된다는 단점이 있다.

---

## 실무 권장 패턴

| 리소스 유형 | 권장 설정 | 이유 |
|------------|----------|------|
| 정적 자산 (파일명에 해시) | `Cache-Control: public, max-age=31536000, immutable` | 내용이 바뀌면 URL이 바뀌므로 무제한 캐싱 안전 |
| HTML 페이지 | `Cache-Control: no-cache` | 진입점이라 항상 최신 자산 참조를 받아야 함 |
| API 응답 (자주 변하는 데이터) | `Cache-Control: no-store` 또는 `max-age=0, must-revalidate` | 데이터 정확성이 우선 |
| API 응답 (잘 안 변하는 데이터) | `Cache-Control: public, max-age=60` 또는 ETag 사용 | 짧은 TTL + 재검사 |
| 사용자별 콘텐츠 | `Cache-Control: private, no-cache` | 다른 사용자에게 노출되면 안 됨 |
| 민감한 정보 (인증, 결제) | `Cache-Control: no-store` | 저장 자체가 위험 |
| 이미지/미디어 | `Cache-Control: public, max-age=86400` + ETag | 적당히 길게, 변경 시 재검사 |

---

## 자주 하는 실수

### 1. no-cache로 캐시 막으려는 실수

`no-cache`는 캐시를 막는 게 아니라 재검사를 강제하는 것이다. 진짜로 막으려면 `no-store`.

### 2. Vary: User-Agent 남용

User-Agent 종류가 수천 가지라 캐시 효율이 0에 수렴한다. 모바일/데스크톱 분기는 서버에서 처리하고 더 단순한 헤더로 분기하자.

### 3. ETag를 약하게 쓰지 않아서 발생하는 비효율

동일 콘텐츠인데 서버 인스턴스마다 다른 ETag를 생성하면 (예: 파일 inode 기반), 로드밸런서 뒤에서 같은 페이지인데도 매번 재전송이 발생한다. 콘텐츠 해시 기반 ETag로 통일하는 게 좋다.

### 4. POST 응답을 캐싱 시도

POST는 기본적으로 캐싱 대상이 아니다. 멱등성이 없어서 캐싱하면 위험하다. 캐싱하려면 GET으로 모델링하자.

### 5. 인증된 응답이 공유 캐시에 저장

`Authorization` 헤더가 있는 요청의 응답은 기본적으로 공유 캐시에 저장되지 않는다. `Cache-Control: public`을 명시해야 저장된다. 이걸 모르고 사용자 A의 응답이 사용자 B에게 노출되는 사고가 종종 발생한다.

---

## 핵심 정리

> 캐시는 원 서버 왕복을 줄이는 가장 강력한 성능 최적화 수단이다. `Cache-Control: max-age`로 신선도를 지정하고, 만료 후에는 `If-Modified-Since`나 `If-None-Match`로 재검사한다. 재검사가 성공하면 본문 없이 `304`만 받아 대역폭을 절약한다. `Vary` 헤더로 헤더별 응답 분기를, `no-cache`(재검사 강제)와 `no-store`(저장 금지)로 정책을 제어한다. 정적 자산은 해시 URL + 긴 max-age + immutable, HTML은 no-cache, API는 짧은 TTL 또는 ETag, 민감 데이터는 no-store가 실무 기본 패턴이다.

# HTTP 완벽가이드 10장 — HTTP-NG와 그 이후

## 들어가기 전에

이 책의 10장은 **HTTP-NG**(HTTP Next Generation)에 대해 다룬다. HTTP-NG는 1990년대 후반에 W3C에서 진행했던 차세대 HTTP 프로토콜 연구 프로젝트다. 결과적으로 HTTP-NG는 표준이 되지 못하고 사실상 사라졌지만, 그 문제의식과 설계 방향은 이후 HTTP/2와 HTTP/3에 큰 영향을 미쳤다.

그래서 이 장은 두 부분으로 정리하는 게 자연스럽다.

1. **책 시점**: HTTP-NG가 무엇을 해결하려고 했는가
2. **현재 시점 (2026)**: HTTP-NG가 미해결로 남겼던 문제들을 HTTP/2, HTTP/3가 어떻게 풀었는가

---

## HTTP/1.1의 한계 — HTTP-NG가 등장한 배경

책이 쓰인 시점에 HTTP/1.1은 이미 자리잡고 있었지만, 여러 한계가 드러나 있었다.

### 1. 복잡성

HTTP/1.1 명세는 너무 복잡하다. 메시지 파싱, 커넥션 관리, 캐시 동작, 콘텐츠 협상 등 모든 것이 한 프로토콜에 뒤섞여 있다. 명세를 정확히 구현하기 어렵고, 구현체마다 미묘하게 다르다.

### 2. 확장성 부족

새 기능을 추가하려면 새 헤더를 정의하거나 기존 동작을 변경해야 한다. 깨끗하게 확장할 수 있는 메커니즘이 부족하다.

### 3. 성능 한계

- 하나의 TCP 커넥션 위로 한 번에 하나의 요청만 처리 가능 (파이프라이닝이 있긴 하지만 한계가 명확함)
- 텍스트 기반 헤더의 오버헤드
- HOL(Head-of-Line) 블로킹 문제

### 4. 보안과 인증의 통합 부족

HTTPS는 HTTP 위에 TLS를 끼워 넣은 형태로, 프로토콜 자체가 보안을 통합적으로 다루지 않는다.

이런 한계들을 근본적으로 해결하기 위해 HTTP-NG가 제안되었다.

---

## HTTP-NG의 설계 사상

HTTP-NG의 핵심은 **HTTP를 모듈화하자**는 것이었다. 하나의 거대한 프로토콜을 세 개의 독립된 계층으로 나누는 구조를 제안했다.

### 1. 메시지 전송 계층 (Message Transport Layer)

가장 아래 계층. 메시지를 효율적으로 전달하는 책임만 진다. 메시지의 의미는 모른다.

- 이진 프레이밍 (텍스트가 아닌 바이너리 메시지)
- 하나의 커넥션 위에서 여러 메시지를 동시에 (멀티플렉싱)
- 효율적인 메시지 식별과 라우팅

### 2. 원격 호출 계층 (Remote Invocation Layer)

중간 계층. 요청-응답 패턴, 비동기 호출, 메서드 호출 같은 추상화를 제공한다. 메시지 계층 위에서 동작하지만, 그 아래 메시지 계층이 어떻게 동작하는지는 신경 쓰지 않는다.

### 3. 웹 애플리케이션 계층 (Web Application Layer)

가장 위 계층. HTTP의 의미론(메서드, 상태코드, 헤더의 의미 등)을 정의한다. GET/POST가 무엇을 의미하는지, 200/404가 무엇인지는 이 계층에서 정해진다.

### 모듈화의 이점

이 분리의 핵심 통찰은 **각 계층을 독립적으로 진화시킬 수 있다**는 점이다. 전송 방식을 바꿔도 위 계층은 영향받지 않고, 의미론을 확장해도 아래 계층은 그대로 둘 수 있다.

---

## HTTP-NG가 풀려고 했던 구체적 문제들

### 멀티플렉싱

하나의 TCP 커넥션 위로 여러 요청-응답을 동시에 처리할 수 있도록 한다. HTTP/1.1은 한 번에 하나씩, 파이프라이닝을 써도 응답 순서가 요청 순서와 같아야 한다는 제약이 있었다.

### 메시지 효율

텍스트가 아닌 이진 프레이밍을 도입해서 파싱이 빠르고 메시지 크기가 작다.

### 흐름 제어

각 요청-응답 스트림마다 독립적인 흐름 제어를 제공한다. TCP의 흐름 제어보다 세밀하다.

### 우선순위

여러 요청 중 어떤 것을 먼저 처리할지 우선순위를 지정할 수 있다.

### 보안 통합

암호화·인증을 프로토콜에 통합적으로 설계한다.

---

## HTTP-NG는 왜 실패했는가

야심찬 설계였지만 실제로는 표준이 되지 못하고 사라졌다. 이유는 다음과 같다.

1. **너무 큰 변화** — 기존 HTTP/1.1 인프라와 호환되지 않아서 도입 부담이 컸다.
2. **즉각적인 이득 부족** — 점진적으로 도입할 방법이 없었다.
3. **HTTP/1.1의 점진적 개선** — 파이프라이닝, 압축, Keep-Alive 등으로 당시의 문제를 어느 정도 완화하면서 시급성이 줄었다.
4. **합의 부족** — 다양한 이해관계자들이 모두 동의할 만한 단일 설계를 만들어내지 못했다.

결국 W3C의 HTTP-NG 활동은 1999년경 사실상 종료되었다.

---

## HTTP-NG 이후 — HTTP/2 (2015)

HTTP-NG가 풀려고 했던 문제들은 **Google의 SPDY 프로토콜**(2009)을 기반으로 한 HTTP/2(RFC 7540, 2015년 표준화)에서 해결되었다. HTTP-NG의 모듈화 사상과 비슷한 분리를 채택했지만, **의미론(메서드·상태코드·헤더)은 HTTP/1.1과 완전히 호환**시켰다는 것이 결정적인 차이다. 기존 웹과의 호환성을 유지하면서 전송 계층만 교체한 것이다.

### HTTP/2의 핵심 변화

#### 1. 이진 프레이밍 (Binary Framing)

HTTP/1.1의 텍스트 메시지를 이진 프레임으로 바꿨다. 파싱이 빠르고 오류가 적다.

```
HTTP/2 프레임 구조:
┌─────────────────────────┐
│ Length (24bit)          │
│ Type (8bit)             │
│ Flags (8bit)            │
│ Stream Identifier (32bit)│
│ Frame Payload (가변)     │
└─────────────────────────┘
```

#### 2. 스트림 (Streams) 개념

하나의 TCP 커넥션 안에 여러 개의 양방향 스트림이 동시에 존재할 수 있다. 각 스트림은 독립된 요청-응답을 처리한다. **HTTP 레벨의 HOL 블로킹이 사라졌다.**

```
TCP 커넥션 1개
├── Stream 1: GET /index.html
├── Stream 3: GET /style.css
├── Stream 5: GET /app.js
└── Stream 7: GET /image.png
```

#### 3. 헤더 압축 (HPACK)

HTTP/1.1에서는 거의 동일한 헤더를 매 요청마다 반복 전송한다(`User-Agent`, `Cookie` 등). HPACK은 정적 테이블과 동적 테이블, 그리고 허프만 인코딩을 조합해서 헤더를 압축한다. 같은 헤더는 인덱스 한 바이트로 줄어든다.

#### 4. 서버 푸시 (Server Push)

서버가 클라이언트의 요청 없이 미리 리소스를 보낼 수 있다. HTML을 보내면서 CSS와 JS를 같이 푸시하는 식. 다만 실제로는 캐시 충돌, 우선순위 문제 등으로 기대만큼 활용되지 못했고, **Chrome은 2022년에 서버 푸시 지원을 제거했다**.

#### 5. 스트림 우선순위

클라이언트가 각 스트림에 우선순위를 부여할 수 있다. 중요한 리소스(HTML, CSS)를 덜 중요한 것(이미지) 보다 먼저 받게 할 수 있다.

#### 6. 흐름 제어

각 스트림과 커넥션 전체에 독립적인 흐름 제어 윈도우가 있다. 빠른 송신자가 느린 수신자를 압도하는 것을 막는다.

### HTTP/2의 한계 — TCP HOL 블로킹

HTTP/2는 HTTP 레벨의 HOL 블로킹은 해결했지만, **TCP 레벨의 HOL 블로킹은 여전히 남았다**. TCP는 본질적으로 순서가 보장된 바이트 스트림이라, 패킷 하나가 손실되면 그 뒤의 모든 패킷은 재전송을 기다려야 한다. 손실된 패킷이 어느 스트림에 속하든 상관없이 다른 스트림까지 영향을 받는다.

이게 HTTP/3의 등장 이유다.

---

## HTTP/3 (2022) — TCP를 버리다

RFC 9114로 2022년 6월에 표준화된 HTTP의 최신 버전이다. 가장 큰 변화는 **TCP를 버리고 QUIC을 사용한다**는 것이다.

### QUIC이란

UDP 위에 만들어진 새로운 전송 프로토콜이다. TCP가 OS 커널에 깊이 박혀 있어서 발전이 느린 반면, QUIC은 사용자 공간에서 동작해서 빠르게 진화할 수 있다.

```
HTTP/1.1, HTTP/2:  HTTP → TLS → TCP → IP
HTTP/3:            HTTP → QUIC (TLS 1.3 내장) → UDP → IP
```

### HTTP/3의 핵심 이점

#### 1. 진정한 멀티플렉싱

QUIC은 각 스트림을 독립된 단위로 처리한다. 한 스트림의 패킷 손실이 다른 스트림에 영향을 주지 않는다. **TCP 레벨의 HOL 블로킹이 완전히 사라졌다.**

#### 2. 빠른 커넥션 수립

TCP + TLS 1.3은 핸드셰이크에 2-3 RTT가 필요하지만, **QUIC은 1-RTT, 재방문 시에는 0-RTT까지 가능하다**. 첫 패킷에 바로 HTTP 요청을 실어 보낼 수 있어서, 모바일에서 페이지 첫 화면 렌더링이 즉시 시작된다.

#### 3. 커넥션 마이그레이션

TCP 커넥션은 `<출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트>` 4-튜플에 묶여 있어서, 와이파이에서 LTE로 전환되면 IP가 바뀌면서 커넥션이 끊긴다. **QUIC은 Connection ID로 커넥션을 식별**하기 때문에, IP가 바뀌어도 커넥션이 유지된다. 모바일 사용자에게 큰 이득이다.

#### 4. TLS 1.3 내장

QUIC은 처음부터 TLS 1.3을 내장해서 설계되었다. 모든 트래픽이 기본적으로 암호화된다.

#### 5. 헤더 압축 (QPACK)

HPACK의 후속. HTTP/2의 HPACK은 헤더 처리 순서가 보장되어야 했는데, QUIC의 비순서적 스트림 처리와 맞지 않아서 QPACK으로 재설계되었다.

### HTTP/3 채택 현황 (2026 기준)

2026년 5월 기준, HTTP/3는 측정 방법에 따라 웹 트래픽의 21~39%를 차지한다. W3Techs는 웹사이트의 39.2%, Cloudflare는 엣지 네트워크의 35%가 HTTP/3를 사용한다고 보고한다. Chrome, Firefox, Safari, Edge 등 모든 주요 브라우저가 기본적으로 HTTP/3를 지원한다.

모바일 중심 시장(이탈리아 30.2%, 브라질 29.3%, 인도 29.1%)에서 채택률이 가장 높다. 손실이 잦은 네트워크에서 QUIC의 이점이 가장 크기 때문이다.

### HTTP/3의 흥미로운 한계

2024년 ACM WWW 컨퍼런스 논문(Zhang et al.)은 1Gbps 환경에서 QUIC이 HTTP/2보다 최대 45.2% 느릴 수 있음을 보였고, 500~600 Mbps 부근에서 HTTP/2가 더 유리해지는 임계점이 관찰된다. 유선 고속 네트워크에서는 HTTP/2가 더 나을 수도 있다는 것이다.

HTTP/3의 가장 큰 이점은 손실이 잦은 네트워크(셀룰러, 모바일, 신흥 시장)에서 30~50%의 개선이 나타나는 영역이다. 모든 환경에서 무조건 빠른 게 아니라, **모바일과 손실 환경에서 강점이 분명한 프로토콜**로 봐야 한다.

### 배포 전략

HTTP/3는 HTTP/2의 대체가 아니라 보완으로 배포해야 한다. Alt-Svc 헤더를 통해 업그레이드 경로로 제공하는 것이 권장된다.

```
Alt-Svc: h3=":443"; ma=86400
```

이 헤더가 있으면 클라이언트는 다음 요청부터 HTTP/3로 시도하고, 실패하면 HTTP/2로 폴백한다.

## 새로운 프로토콜들 — QUIC 생태계 (2026)

HTTP/3의 등장으로 QUIC을 기반으로 한 새로운 프로토콜들이 등장했다.

### WebTransport

WebSocket의 후속이라고 할 수 있는 양방향 통신 프로토콜이다. QUIC 위에서 동작하며, 신뢰성 있는 스트림과 비신뢰성 데이터그램을 모두 지원한다. 게임, 라이브 비디오, 실시간 협업 같은 저지연 양방향 통신에 적합하다.

### HTTP/3 Datagrams + MASQUE

QUIC의 데이터그램 기능을 활용해서 다양한 프로토콜을 HTTP/3 위로 터널링할 수 있다. VPN, 프록시 서비스가 별도 클라이언트 없이 HTTP/3 인프라를 활용해 동작할 수 있게 된다.

### 0-RTT의 보안 고려사항

0-RTT는 빠르지만 재전송 공격(replay attack)에 취약하다. 멱등 메서드(GET)만 0-RTT로 보내고, POST 같은 비멱등 메서드는 1-RTT 이후에 보내는 것이 안전 패턴이다.

## 핵심 정리

> HTTP-NG는 HTTP를 세 계층으로 모듈화하려 했던 1990년대 후반의 시도였고, 표준이 되진 못했지만 그 문제의식은 HTTP/2(2015)에서 부활했다. HTTP/2는 이진 프레이밍·멀티플렉싱·헤더 압축으로 HTTP 레벨의 HOL 블로킹을 해소했지만 TCP HOL 블로킹은 남았다. HTTP/3(2022)는 TCP를 버리고 QUIC(UDP 기반)을 채택해서 TCP HOL 블로킹까지 해결했고, 0-RTT 핸드셰이크와 커넥션 마이그레이션을 통해 특히 모바일에서 강점을 보인다. 2026년 기준 HTTP/3는 트래픽의 35~40%를 차지하지만, 모든 환경에서 빠른 건 아니며 손실 잦은 모바일 네트워크에서 가장 효과적이다. 모놀리식 백엔드 입장에서는 CDN/리버스 프락시 레이어에서 HTTP/3를 종료하는 패턴이 가장 실용적이다.
